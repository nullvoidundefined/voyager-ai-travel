## Executive Summary

Overall risk posture: **Moderate, improved from the 2026-04-06 baseline.** The major P0 and P1 findings from April are resolved: the photo proxy is now authenticated, the dependency tree has no high CVEs in the server package (the CI gate enforces this), the column-injection path in `updateTrip` is closed, and the SDK was upgraded to patch the path-traversal CVE. The agent safety limits (8 tool calls, 120s wall-clock) are enforced.

Three new findings require immediate attention.

**Top 3 findings today:**

1. **P0 -- Real database connection string in `.env.example` committed to git history.** `server/.env.example` contains an active (non-placeholder) `DATABASE_URL` value that was committed in the initial scaffold commit `15bc653` and remains in the public git history. The value was flagged and redacted by the secret scanner on two independent reads. This is a credential in the permanent git record. The Neon database connection string must be rotated immediately.

2. **P1 -- Next.js 15.5.14 has 8 high-severity CVEs, all patched in 15.5.16-15.5.18.** The vulnerabilities include middleware/proxy authentication bypass (GHSA-3g8h-86w9-wvmq, GHSA-mg66-mrh9-m8jx, GHSA-c4j6-fc7j-m34r), SSRF via WebSocket upgrades, and two Server Component DoS vectors. No Next.js middleware is in use in Voyager, which reduces the middleware-bypass severity, but the SSRF and DoS vectors apply. The fix is a single version bump.

3. **P1 -- `@anthropic-ai/sdk` 0.81.0 has a moderate CVE (GHSA fixed in 0.91.1).** The advisory covers insecure default file permissions in the local filesystem memory tool. Voyager does not use the filesystem memory tool, which limits exploitability to theoretical, but the SDK is on a version that predates 10 weeks of patches. Upgrade to `>=0.91.1`.

**Credentials requiring action:**

- `DATABASE_URL` in `server/.env.example` and git history: **rotate immediately**. The Neon connection string should be revoked in the Neon dashboard and a new one issued. Do not issue a replacement until the commit history is either scrubbed (force-push + contact GitHub support) or the repo is confirmed private.
- `NEXT_PUBLIC_GOOGLE_MAPS_KEY`: this key is embedded in client-side Static Maps API URLs (`MapPreviewCard.tsx:36`) and is therefore visible in browser network tabs and page source. Confirm it has HTTP referrer restrictions scoped to `interviewiangreenough.xyz` in Google Cloud Console. If not restricted, restrict or rotate it.
- No new `sk-ant-*` or SerpApi key exposure detected in git history, session transcripts (4 JSONL files scanned, 0 matches), or shell history.

## Threat Model

**Trust boundaries:**

- Public internet -> Vercel edge (Next.js, `/api/*` rewrite proxy) -> Railway Express API -> Neon Postgres + Redis
- Browser -> `/places/photo` (authenticated) -> Google Places API (paid key)
- Authenticated user -> `/trips/:id/chat` -> Anthropic API (paid key) -> SerpApi (paid quota)
- CI runner (GitHub Actions) -> Docker build -> Railway deploy

**Attacker personas:**

1. **Unauthenticated external attacker** scanning the public surface. No registered account. Goals: enumerate users, harvest credentials, exploit client-side vulnerabilities.
2. **Authenticated low-privilege user** (registered account). Goals: access another user's trips (IDOR), exhaust paid API quota (prompt injection), escalate to admin functionality.
3. **Supply-chain attacker** targeting npm packages or CI secrets.

**Attack trees (top 3):**

1. Rotate the now-exposed `DATABASE_URL` -> connect to Neon directly -> dump `users` table (bcrypt hashes + emails), `sessions` table (SHA-256 token hashes, not reversible), `trips`, `conversations` (PII).
2. Craft user message to `/trips/:id/chat` -> inject into `search_experiences.categories[]` (no allowlist on this field) -> Google Places `textQuery` constructed from attacker-controlled category strings -> scrape arbitrary Places data using Voyager's paid key.
3. Exploit Next.js 15.5.14 SSRF via WebSocket upgrade path -> reach internal Railway service addresses not meant to be externally accessible.

**Blast radius:** If the `DATABASE_URL` in git has already been accessed by a third party, all user emails, bcrypt hashes, trip PII, and conversation history are potentially exfiltrated. Session tokens are not directly recoverable (SHA-256 hashed in the DB), but a DB dump enables offline bcrypt cracking.

## Authentication and Session Management

**Still correct (unchanged from April):**

- bcrypt at 12 rounds (`SALT_ROUNDS = 12`). Correct.
- Session tokens: 32 random bytes, cookie holds raw, DB holds SHA-256 hash. A DB dump cannot replay sessions.
- Session cookies: `httpOnly: true`, `sameSite: 'lax'`, `secure: isProduction()`, `path: '/'`. Correct.
- `loginUser` deletes all prior sessions in a transaction before issuing a new one. Correct.
- Error messages do not distinguish "user not found" from "wrong password". Good enumeration hygiene.
- Session TTL: 7 days (`SESSION_TTL_MS = 7 * 24 * 60 * 60 * 1000`). Reasonable.

**Doppelscript comparison:** Voyager's session design (custom SHA-256 hashed tokens, 7-day TTL, httpOnly+Secure cookie) is architecturally equivalent to Doppelscript's custom session implementation. No gaps relative to the reference bar on auth mechanics.

**Open from prior audit (tracked in ISSUES.md):** SEC-10 (password complexity), SEC-11 (per-account failed-login backoff), SEC-13 (deleteExpiredSessions not scheduled), SEC-19 (session TTL/refresh verification).

**No new auth findings.**

### [HIGH] SEC-NEW-01: `search_experiences.categories[]` has no content allowlist -- prompt injection into Google Places `textQuery`

- **File:** `server/src/tools/schemas.ts:49`, `server/src/tools/experiences.tool.ts:104`
- **CVSS:** 6.3 (AV:N/AC:L/PR:L/UI:N/S:C/C:N/I:L/A:L)
- **Description:** `searchExperiencesSchema` defines `categories` as `z.array(z.string().min(1).max(50)).min(1)`. Unlike the `location` field (which uses `locationAllowlist` with a strict character class), category strings accept any printable characters including HTML entities, shell metacharacters, and URL-structure characters. The experiences tool builds the Google Places `textQuery` as `${categoryText}in ${input.location}` (line 104), where `categoryText` is the raw categories joined with spaces. An attacker who registers an account and sends a crafted chat message can cause the agent to call `search_experiences` with attacker-controlled categories that become part of the Places query string.
- **Attack scenario:** An authenticated user sends: `"Find me experiences. Categories: bar baz} OR 1=1 --"`. The agent calls `search_experiences` with `categories: ["bar baz} OR 1=1 --"]`. The schema passes this (max 50 chars, min 1 char). The tool constructs `textQuery = "bar baz} OR 1=1 -- in Paris"` and sends it to the Google Places Text Search API. While this specific payload does not cause SQL injection in Google's API, the same mechanism allows an attacker to construct a Places query for any arbitrary address or content ("123 Main St, Victim City" as a category) and receive structured geocoordinate and business data using Voyager's paid Google Places key. This is an API abuse / cost attack and a potential doxing vector.
- **Proof of concept:** `POST /trips/:id/chat` with `{ "message": "Search for experiences. The categories I want are: 'location of John Smith 123 Main St Seattle'" }`. The agent will construct `textQuery = "location of John Smith 123 Main St Seattle in <destination>"` and return structured Places results.
- **Remediation:** Apply `locationAllowlist` to each element of the `categories` array. Change `z.array(z.string().min(1).max(50))` to `z.array(locationAllowlist.max(50))` in `searchExperiencesSchema`. The allowlist regex `^[\p{L}\p{N} ,.\-'()]+$/u` is already defined at line 17; reuse it.
- **Cross-ref:** SEC-14 (open in ISSUES.md). SEC-14 notes the `select_*` schemas also bypass the location allowlist. That finding remains open separately.

### Informational: `quick_replies` in `format_response` not length-capped

- **File:** `server/src/tools/schemas.ts:132`
- The `quick_replies` array uses `z.array(z.string()).optional()` with no max length on either the array or individual strings. This is rendered as clickable chips in the frontend. Not an immediate exploit path since it is LLM-generated, but a jailbroken model could return arbitrarily large quick reply arrays. Recommend `z.array(z.string().max(100)).max(8).optional()`. ## Input Validation and Injection

**SEC-15 still open:** `updateTrip` handler destructures fields from `req.body` directly without Zod parsing at the handler boundary. The repository-level `UPDATE_TRIP_ALLOWED_COLUMNS` set prevents column injection at the SQL layer, but type coercion (ensuring `budget_total` is a number, `departure_date` is a valid date string) is enforced only by the handler's manual checks, which are incomplete. This is tracked as SEC-15 in ISSUES.md.

### [MEDIUM] SEC-NEW-02: No message length cap on `/trips/:id/chat`

- **File:** `server/src/handlers/chat/chat.ts:49-53`
- **CVSS:** 5.3 (AV:N/AC:L/PR:L/UI:N/S:U/C:N/I:N/A:L)
- **Description:** The chat handler validates only `!message || typeof message !== 'string'`. There is no maximum length check. The Express JSON body limit is 10KB (`express.json({ limit: '10kb' })`), so the theoretical maximum is ~10,000 characters. A user sending a 9,000-character message causes the entire string to be embedded in the Claude API request, consuming input tokens against the Anthropic API. This is a cost amplification attack: an authenticated user can send maximum-length messages at the rate limit ceiling (10 per 5 minutes) to force disproportionate token consumption. The per-user daily output token budget does not cap input tokens.
- **Attack scenario:** Authenticated user sends 10 requests of 9,000 characters each within the 5-minute window. Each triggers a Claude API call with a bloated context. Input tokens are explicitly not counted against the daily budget (chat.ts:158 comment). This costs the operator money and cannot be rate-limited below the 10/5m chat ceiling without additional per-message input-token checking.
- **Remediation:** Add a `MAX_MESSAGE_LENGTH` constant and reject messages exceeding it before the SSE stream opens:
  ```typescript
  const MAX_MESSAGE_LENGTH = 2000;
  if (message.length > MAX_MESSAGE_LENGTH) {
    throw ApiError.badRequest(
      `Message must be ${MAX_MESSAGE_LENGTH} characters or fewer`,
    );
  }
  ```
  2000 characters covers all legitimate travel planning messages and eliminates the token-stuffing attack surface.

## CSRF and Cross-Origin

**Current implementation:** `csrfGuard` checks for the presence of `X-Requested-With` on state-changing methods. The frontend unconditionally sends `X-Requested-With: XMLHttpRequest` on all requests (`web-client/src/lib/api.ts:26`). CORS is enforced with an explicit allowlist.

**Still open (SEC-06):** The CSRF guard is a pure header-presence check with no cookie-bound entropy. Any non-browser client (curl, a server-side script) can trivially supply `X-Requested-With: XMLHttpRequest`. The Doppelscript reference uses the same pattern, so this is not a regression against the quality bar.

**Logout is not rate-limited.** `POST /auth/logout` has no `authRateLimiter`. An unauthenticated client can call it 100 times per 15 minutes (global limiter budget). Logout is idempotent (it deletes the session if present, no-ops if not), so this is not exploitable for session destruction without the session cookie. Low risk.

**No new CSRF findings.**

### [LOW] SEC-NEW-03: Test seam route `/:id/test-selections` registered unconditionally in `routes/trips.ts`

- **File:** `server/src/routes/trips.ts:23`, `server/src/handlers/trips/trips.ts:258-262`
- **CVSS:** 2.6 (AV:N/AC:H/PR:L/UI:N/S:U/C:N/I:L/A:N)
- **Description:** `tripRouter.post('/:id/test-selections', tripHandlers.seedSelections)` is registered in production. The handler returns 404 unless `E2E_BYPASS_RATE_LIMITS === '1'`, which is not set in production. However, the route is discoverable by scanners, the handler is loaded, and its ownership check (`getTripWithDetails(tripId, userId)`) runs before the env flag check is evaluated. If `E2E_BYPASS_RATE_LIMITS` were accidentally set in a production deployment (e.g., a misconfigured env var), this route would allow an authenticated user to write arbitrary unvalidated flight/hotel/car_rental/experience records to any trip they own, bypassing the Zod validation that the real `selectItem` handler applies.
- **Attack scenario:** An operator accidentally sets `E2E_BYPASS_RATE_LIMITS=1` in Railway. An authenticated user calls `POST /trips/:id/test-selections` with malformed or extreme numeric values (e.g., `price: 999999999`) for flights/hotels. These bypass selectFlightSchema/selectHotelSchema validation and are inserted directly via the repo layer.
- **Remediation:** Move the env check to the top of the handler before the ownership query (already mostly done), and additionally gate route registration itself:
  ```typescript
  if (process.env.NODE_ENV !== 'production') {
    tripRouter.post('/:id/test-selections', tripHandlers.seedSelections);
  }
  ```
  Or use a dedicated `E2E_ROUTES_ENABLED` env var distinct from `E2E_BYPASS_RATE_LIMITS` so the test seam cannot be accidentally activated.

## Secrets Management

### [CRITICAL] SEC-NEW-04: Real database connection string committed to `server/.env.example` in git history

- **File:** `server/.env.example:5` (current working tree), git commit `15bc653`
- **CVSS:** 9.8 (AV:N/AC:L/PR:N/UI:N/S:U/C:H/I:H/A:H)
- **Description:** The `DATABASE_URL` field in `server/.env.example` contains what the `redact-output.sh` secret scanner identified as a real credential value (not the placeholder `postgresql://user:password@host:5432/dbname`). The value was redacted on two independent reads by the PostToolUse hook. The commit `15bc653` ("Initial scaffold from template-express-next with travel agent DB schema") added this file. It is in the permanent git history of this repository. If the repository is or has ever been public, or if the history has been cloned, the Neon connection string is exposed.
- **Attack scenario:** An attacker clones or browses the repository. They read `server/.env.example` in the initial commit. They extract the `DATABASE_URL`. They connect to the Neon Postgres instance and issue `SELECT id, email, first_name, last_name FROM users` and `SELECT * FROM trips`. Full user PII and travel history is exposed. Sessions cannot be replayed (SHA-256 hashed), but email/bcrypt hash pairs enable offline dictionary attacks.
- **Proof of concept:** `git show 15bc653:server/.env.example | grep DATABASE_URL`.
- **Remediation:**
  1. **Immediately:** Rotate the Neon database credential in the Neon dashboard. Issue a new connection string. Update the Railway environment variable.
  2. Replace the current `server/.env.example` `DATABASE_URL` value with the literal placeholder `postgresql://user:password@host:5432/dbname`.
  3. Commit the sanitized `.env.example` to purge the credential from the working tree.
  4. Contact GitHub support or use `git filter-repo` to scrub the credential from git history. Note: `git filter-repo` rewrites history and requires a force-push. Until history is scrubbed, treat the credential as permanently exposed.
  5. Enable Neon's audit log and check for unexpected connection attempts since the initial commit date.

### [HIGH] SEC-NEW-05: `NEXT_PUBLIC_GOOGLE_MAPS_KEY` embedded in client-side Static Maps URLs without confirmed referrer restriction

- **File:** `web-client/src/components/ChatBox/widgets/MapPreviewCard.tsx:36`
- **CVSS:** 5.3 (AV:N/AC:L/PR:N/UI:N/S:U/C:L/I:N/A:N)
- **Description:** `MapPreviewCard.tsx` constructs a Google Static Maps URL with `&key=${apiKey}` where `apiKey = process.env.NEXT_PUBLIC_GOOGLE_MAPS_KEY`. This key is shipped to the browser (all `NEXT_PUBLIC_*` values are inlined at build time and visible in page source and network requests). The Maps Static API key in the URL is readable by any user who opens browser DevTools or views page source. If this key is not restricted by HTTP referrer or API scope in Google Cloud Console (and SEC-21 in ISSUES.md flags that this has not been verified), any third party can use the key to make unlimited Static Maps API requests at Voyager's billing expense.
- **Attack scenario:** User opens DevTools on a trip detail page, copies the Static Maps URL and its `key=` parameter. They script 10,000 Static Maps requests per day using that key. Voyager is billed for all requests. Google's Maps Static API pricing is $2/1000 requests, so 100,000 requests = $200.
- **Remediation:** In Google Cloud Console, restrict `NEXT_PUBLIC_GOOGLE_MAPS_KEY` to:
  1. HTTP referrer: `https://interviewiangreenough.xyz/*` (and Vercel preview pattern if applicable).
  2. API scope: Maps Static API only. Not the same key as `GOOGLE_PLACES_API_KEY` (which is a server-side key and must not be referrer-restricted since it is called from Railway, not a browser).
     Verify the two keys are separate. If they are the same key, split them immediately and rotate.
- **Cross-ref:** SEC-21 (ISSUES.md).

### [HIGH] SEC-NEW-06: Next.js 15.5.14 has 8 high-severity CVEs patched in 15.5.16-15.5.18

- **File:** `web-client/package.json` (`"next": "15.5.14"`)
- **CVSS:** 7.5 (AV:N/AC:L/PR:N/UI:N/S:U/C:N/I:N/A:H) for the DoS vectors; 8.1 (AV:N/AC:H/PR:N/UI:N/S:C/C:H/I:H/A:N) for the SSRF
- **Relevant CVEs:**
  - GHSA-3g8h-86w9-wvmq: Middleware/proxy bypass via segment-prefetch routes (patched in 15.5.16)
  - GHSA-mg66-mrh9-m8jx: Middleware/proxy bypass via dynamic route parameter injection (patched in 15.5.16)
  - GHSA-c4j6-fc7j-m34r: Middleware/proxy bypass in Pages Router via i18n (patched in 15.5.16)
  - GHSA-vfv6-92ff-j949: SSRF via WebSocket upgrade (patched in 15.5.16)
  - GHSA-8h8q-6873-q5fj / GHSA-q4gf-8mx6-v5v3: Server Component DoS (patched in 15.5.15/15.5.16)
  - GHSA-36qx-fr4f-26g5: Middleware/proxy bypass (incomplete fix, patched in 15.5.18)
  - GHSA-267c-6grr-h53f: Server Component DoS via connection exhaustion (patched in 15.5.16)
- **Description:** Voyager does not use Next.js Middleware (`middleware.ts` not found), which reduces the middleware-bypass CVEs from critical to high. The SSRF via WebSocket upgrade path (`GHSA-vfv6-92ff-j949`) applies to any Next.js application using the rewrite proxy feature, which Voyager does (`next.config.ts` rewrites `/api/:path*` to the Express server). An attacker may be able to craft a WebSocket upgrade request that causes Next.js to proxy to an internal Railway address.
- **Attack scenario (SSRF):** Attacker sends a WebSocket upgrade request to the Vercel-hosted Next.js app targeting an internal Railway private network address through the `/api/` rewrite proxy. Next.js 15.5.14 has a known path where WebSocket upgrade handling does not properly restrict the destination. The attacker reaches internal services on the Railway private network that are not meant to be externally accessible.
- **Remediation:** Bump `"next"` in `web-client/package.json` to `"^15.5.18"` (the latest patched version as of audit date that addresses all known issues). Run `pnpm install --frozen-lockfile=false` to update the lockfile. Run `pnpm test:e2e` to verify no regressions. Extend the CI `pnpm audit` gate to cover `web-client` in addition to the server package.

### [MEDIUM] SEC-NEW-07: `@anthropic-ai/sdk` 0.81.0 has a moderate CVE (insecure file permissions in filesystem memory tool)
