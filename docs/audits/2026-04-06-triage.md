### [FIN-01] No hard monthly cap on Anthropic

- **Source:** financial §Service Inventory, Anthropic, §Spending Caps In Place
- **Severity:** P0 · **Effort:** S · **Category:** financial / operational
- **Repro:** inspect Anthropic console for the project org; no monthly spend cap configured; no per-user token budget in code
- **Fix approach:** set a hard $50/mo cap in the Anthropic console today. Add a `rate-limit.service.ts` per-user daily token counter (Redis) that blocks `orchestrator.run()` when the user has exceeded a daily output-token budget. Lower `DEFAULT_MAX_ITERATIONS` from 15 to 8 until observability is in place.

### [FIN-02] No SerpApi monthly counter or spending cap

- **Source:** financial §Service Inventory, SerpApi
- **Severity:** P0 · **Effort:** M · **Category:** financial
- **Repro:** `server/src/services/serpapi.service.ts` has no monthly counter; at search 251 the feature silently breaks; agent degrades to hallucinated flight data
- **Fix approach:** implement a Redis-backed monthly counter keyed by `YYYY-MM`. Refuse to call past 200 (reserve 50 for emergencies). Surface a graceful "flight search temporarily unavailable" message to the user. Extend cache TTL from 1h to 6-12h.

### [FIN-03] No Google Cloud billing cap on Places project

- **Source:** financial §Service Inventory, Google Places
- **Severity:** P0 · **Effort:** S · **Category:** financial
- **Fix approach:** set a hard $50/mo billing cap in Google Cloud Console on the Places project.

### [SEC-01] Unauthenticated /places/photo proxy enables cost DoS on Google Places key

- **Source:** security §Input Validation, §Attack tree #1, §P0
- **Severity:** P0 · **Effort:** S · **Category:** security
- **Repro:** `server/src/handlers/places/photoProxy.handler.ts` lines 7-36, `server/src/routes/places.ts` has no `requireAuth` on the route. The `ref` query param flows directly into the Google Places URL. Any anonymous visitor can drain the paid `GOOGLE_PLACES_API_KEY` quota.
- **Fix approach:** add `requireAuth` middleware to the route. Add a strict regex allowlist for `ref`. Clamp `maxwidth`. Add a dedicated rate limiter separate from the global one. Cache responses. After the fix ships, audit the GCP billing console for any evidence of prior abuse. Rotate `GOOGLE_PLACES_API_KEY` if any anomalous spike appears in the last 30 days.
