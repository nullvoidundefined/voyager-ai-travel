# Voyager UX Audit

**Date:** 2026-05-27
**Auditor:** Chief Experience Officer (canonical role: `~/.claude/audits/on-request/ux.md`)
**Scope:** Voyager web client, user stories US-1 through US-35, chat agent UX, booking flow, preferences wizard, a11y, mobile, destructive action guardrails.
**Prior audit:** 2026-04-06-ux.md (referenced throughout for delta tracking)
**Reference standard:** Doppelscript (established UX/a11y baseline per prompt)

---

## Executive Summary

**Can a new user complete the primary happy path (register, preferences, create trip, chat with agent, confirm booking) without help?**

**Yes, with caveats.** Since the April 2026 audit, two P0 blockers have been resolved: trip deletion now uses a Radix AlertDialog with a proper confirmation step, and the BookingConfirmation modal has been redesigned into a simple one-step "Save itinerary" / "Make changes" pattern with a visible demo disclaimer. The worst UX failure modes from the prior audit are gone. A first-time user can now complete the full happy path.

However, several meaningful problems remain:

**Top 3 priorities (P1 / P2):**

1. **P1. US-5 destination pre-fill is silently broken.** The destination detail page (`explore/[slug]/page.tsx:249`) generates a link to `/trips/new?destination=Paris` but `trips/new/page.tsx` never reads `useSearchParams` and always seeds the trip with the literal string `"New trip"`. The query param is silently discarded. This breaks the documented acceptance criteria for US-5 ("After login, redirected to trip creation with destination pre-filled") and violates the user's expectation set by the CTA text "Plan a trip to Paris."

2. **P1. PreferencesWizard and BookingConfirmation are hand-rolled modals without focus traps.** Neither component uses `focus-trap-react` or a Radix Dialog primitive. The Radix AlertDialog was added for trip deletion but the two highest-traffic modals still leak keyboard focus to the page underneath. For screen reader users, tabbing out of either modal silently moves focus to background page elements with no indication the modal is still open. This is a WCAG 2.1 SC 2.1.2 (no keyboard trap) failure and a Lighthouse a11y gap.

3. **P1. AuthGuard redirect loses the intended destination (US-12 partial failure).** `AuthGuard.tsx` always redirects to `/login` with no `?next=` or `?from=` parameter. A user who clicks "Plan a trip to Paris" from the destination page gets redirected to `/login`, logs in, and lands on `/trips` or `/trips/new` (the login page checks their trip count), never returning to the destination that prompted them to sign up. The US-12 acceptance criterion says "After login, redirected back to the original page." This is not implemented.

## Critical Path Analysis: Signup, First Value, Paid Action

**Step 1: Signup.** Register page is clean. Form has proper `autoComplete` attributes (`given-name`, `family-name`, `email`, `new-password`). Error handling uses Toast for API errors and inline `<p className={styles.error}>` for client-side validation. The step indicator (two dots with a connecting line) communicates that registration is multi-step. Labels are present on all fields.

**One issue:** the `<input type="password">` has no password strength indicator. The only feedback is a min-8-chars client-side error after submission. This is a friction point: users enter a short password, hit submit, read the error, try again. A real-time character counter or strength bar would eliminate this round trip. This was not flagged in the prior audit.

**Step 2: Preferences wizard.** UX-16 from the prior audit (no "why we ask" explanation) is still unresolved and remains in ISSUES.md. The wizard opens immediately after registration over a page the user has barely seen. Six preference questions covering accommodation style, travel pace, dining, activities, travel party (including LGBTQ+ safety and gender identity), and budget comfort are asked before the user has seen the product deliver any value. The introductory text "Your Travel Preferences" with a step counter offers no explanation of what these preferences are used for or how they affect trip planning. Per Doppelscript's standard: preferences pages must explain their purpose before asking personal questions. The prior audit found this and it has not been addressed.

**Step 3: Create trip.** `trips/new/page.tsx` now seeds with `"New trip"` instead of `"Planning..."`. The spinner + "Setting up your trip..." message is still the best loading state in the app. No regression here, but the destination placeholder `"New trip"` still shows on the trips list card (`<h2>{trip.destination}</h2>`) until the agent updates it.

**Step 4: First value (chat).** The chat empty state in `VirtualizedChat.tsx:146-153` renders a plane emoji, "Start planning your trip", and "Describe where you want to go, your dates, and budget below." This is a substantial improvement over the prior audit's blank state. The first-time user is coached on how to begin.

The TripDetailsForm shows after sending any initial message, presenting structured fields for origin, departure date, return date, and budget. This is the primary "first value" moment. It works. No regression.

**Step 5: Paid action (confirm booking).** The prior P0 is resolved. The BookingConfirmation modal:

- Has `role="dialog"`, `aria-modal="true"`, `aria-label="Booking confirmation"`.
- Renders an itemized breakdown, total, budget comparison.
- Has a clear demo disclaimer: "Nothing is actually booked."
- Has a direct "Save itinerary" confirm and a "Make changes" cancel, both always visible.
- No animation sequence, no auto-advance timer, no countdown. One click confirms.

This is a clean resolution of the prior P0-2. The action is clear, the consequences are explained, and cancellation is available until the moment of confirmation.

## Error Recovery

Every failure mode a user could hit:

- **SSE stream error.** `useSSEChat.ts:78-84`: sets `error` state. `ChatBox.tsx:223` renders `<Toast message={sseError} onClose={clearSseError} />`. Error message is actionable: "The agent ran into a problem mid-response. Try sending the message again." / "Could not reach the agent. Check your connection and try again." **Fixed from prior audit.** Prior P1 (raw error leaked into `streamingText`) is gone. However: there is no retry button in the Toast. The user has to know to retype their message. The Toast dismisses after 5 seconds (default). If the user is not watching when the Toast appears, they lose the error context. **P2 gap: add a "Try again" button inside the Toast component when used for SSE errors, or keep the Toast open until dismissed.**

- **Network failure creating a new trip.** `trips/new/page.tsx:33-35`: shows "Failed to start a new trip. Please try again." with a back link to `/trips`. No retry button. User must navigate back and click "+ New Trip" again. **P3: add a "Try again" button inline.**

- **Trip detail load failure.** `trips/[id]/page.tsx:119-127`: shows "Trip not found." for both 404 and 500 errors. **UX-15 from prior audit is still unresolved.** Misleading for server errors. Back link is provided.

- **Trip not found vs. server error.** Same issue as above. The query error from React Query does not distinguish HTTP status codes. A 500 should say something like "Could not load your trip. Try refreshing the page." **P2 (carried from UX-15).**

- **User changes their mind mid-plan.** Tile card selection is still reversible only via chat (UX-17, prior audit). No "unselect" affordance on a confirmed tile. **P2 (carried from UX-17).**

- **User deletes a trip by accident.** AlertDialog with "Delete trip" / "Keep trip" options. Full trip/conversation/itinerary consequence is described in the dialog body. No undo. The prior P0 is resolved. No regression.

- **Booking failure.** `trips/[id]/page.tsx:100-103`: if the `PUT /trips/:id` status update fails, a Toast is shown ("Failed to save your booking. Please try again.") and the modal closes. Actionable. Good.

- **Forgot password.** `login/page.tsx:97-99`: clicking "Forgot password?" shows a Toast: "Password reset is coming soon." The feature is not implemented. The FAQ (`faq/page.tsx`) does not mention that password reset is unavailable. For a portfolio demo this is acceptable, but the Toast message is honest.

## Forms and Input

- **Login form:** `autoComplete="email"` and `autoComplete="current-password"` are set. Labels are `<label>` wrappers (implicit association). Error shown via Toast for API errors and inline `<p>` for empty-field validation. The inline error does not have `role="alert"` so screen readers will not announce it on injection. **P2 a11y gap: add `role="alert"` to `<p className={styles.error}>` in login and register pages.**

- **Register form:** Same pattern. `autoComplete` attributes correct. The form is inside a `<label>` wrapper for each input (implicit association). No `aria-describedby` linking the password input to the min-8-chars hint. **P2: add `aria-describedby` pointing to the hint text.**

- **TripDetailsForm:** Proper `<label htmlFor>` / `id` pairings for all fields (`id="origin"`, `id="departure_date"`, etc.). Uses native `<input type="date">` for dates, which is best practice for mobile. Budget has a `$` prefix visually but the input does not have `aria-label` describing the currency unit. A screen reader user hears "Budget" then a number input, not "Budget in US dollars." **P2 a11y gap.**

- **PreferencesWizard step forms:** Each step component (`AccommodationStep`, `TravelPaceStep`, etc.) was not read in full, but the options appear to be presented as button groups (selectable cards). Verify each button has a meaningful text label (not icon-only). Verify the current selection is communicated via `aria-pressed` or `aria-checked`.

- **DOM-scraping auto-save in ChatBox (UX-13):** The prior audit flagged `document.getElementById()` usage for form auto-save. This has been replaced with a ref callback pattern (`formValuesRef` / `handleFormValuesChange`). The fix is in place. **Prior P2 resolved.**

- **The "trip_type" dead state (CQS-30):** `TripDetailsForm` renders Round Trip / One Way toggle buttons and conditionally hides the return date field when "One Way" is selected. However, the `handleSubmit` in `TripDetailsForm` builds a natural-language message from the field values and never includes trip type. `ChatBox.handleFormSubmit` also does not read `structuredData.trip_type`. The toggle has UX meaning (it hides `return_date`) but the selected value is never sent to the API or the agent. A user who selects "One Way" will have their return date field hidden and their form submitted without a return date, which may confuse the agent. This is inherited from CQS-30 in ISSUES.md but has a direct UX impact. **P2: either remove the One Way toggle until it is wired to the API, or pass `trip_type` through `structuredData` in the form submission.**

## Feedback and State Communication

- **Loading states:** `trips/new` has spinner + "Setting up your trip..." (good). `trips/[id]` shows `<p>Loading trip...</p>` with no spinner (minimal). `trips` shows `<p className={styles.loading}>Loading trips...</p>` (minimal). **UX-09 from prior audit remains unresolved** in ISSUES.md. The inconsistency is still present.

- **Agent tool-call transparency:** `VirtualizedChat` renders a `ChatProgressBar` in `determinate` mode showing "X of Y done" and a label like "Searching flights". This is an improvement over the prior audit's `ToolProgressIndicator`. The label updates as tools complete. **UX-11 from prior audit** (no summary of what the agent found) is still unresolved. Once all tools are done, the progress bar disappears and the tile cards appear. There is no collapsed summary node explaining "Found 42 flights, picked 3 under $450." This remains a black-box moment in an agentic product.

- **Empty states:** Chat empty state is good (emoji + title + subtitle). Trips list empty state is good ("No trips yet. Start planning your first adventure!" + New Trip CTA). Trip detail "no itinerary items" empty state is minimal ("No itinerary items yet. Use the chat to start planning!"). The trips list error state shows a red error string with no retry button. **P3: add a retry button or "refresh" link to the trips list error state.**

- **Toast component:** Auto-dismisses after 5 seconds. Has a manual dismiss button (`aria-label="Dismiss"`). The Toast element itself has `className={styles.toast}` with no `role="alert"` or `aria-live="assertive"`. Screen readers will not announce Toast messages. **P1 a11y finding: add `role="alert"` to the Toast's root div.** This affects SSE errors, booking failure messages, and any other system notification surfaced via Toast.

- **Send button state:** Shows "..." while `isSending`. No `aria-label` update during sending state. A screen reader user who activates the send button will not be notified that the message was sent and that the agent is working. **P2 a11y gap: add `aria-busy="true"` to the chat form during sending, or add an `aria-live` region that announces when the agent starts and finishes.**

- **Streaming text:** The deliberate flash-gap prevention fix documented in `useSSEChat.ts` (invalidate queries before clearing streaming state) is present and working. Good.

## Accessibility

**A full Lighthouse run is required before sign-off. The following findings are from code inspection only.**

### Resolved from prior audit

- BookingConfirmation now has `role="dialog"`, `aria-modal="true"`, `aria-label`.
- Chat error is surfaced via Toast instead of raw inline text.
- `prefers-reduced-motion` global override is implemented in `globals.scss` with `!important` on all animation durations and transition durations. All animations across the app are collapsed to 0.01ms for users who have this preference set.
- The DOM-scraping auto-save in ChatBox was replaced with a ref callback.

### New and remaining findings

**P1. Toast has no `role="alert"`.** `Toast.tsx:20` renders `<div className={styles.toast}>`. There is no `role="alert"` or `aria-live` attribute. Any error or status message surfaced via Toast (SSE errors, booking failure, trips load error, password reset notice) is invisible to screen readers. Per Doppelscript's standard, this is a P1 a11y violation because Toast is the primary mechanism for server error communication in the app.

**P1. PreferencesWizard has no focus trap.** `PreferencesWizard.tsx` uses `role="dialog"` and `aria-modal="true"` but there is no focus trap implementation. Keyboard users can Tab past the modal boundary into background page elements. This violates WCAG 2.1 SC 2.1.2. The `onKeyDown` handler at line 194 catches Escape to close, but does not prevent focus from leaving the dialog. Radix Dialog or `focus-trap-react` would fix this.

**P1. BookingConfirmation has no focus trap.** Same issue as PreferencesWizard. The component has `role="dialog"`, `aria-modal="true"`, and `aria-label`, but no focus trap. The Radix AlertDialog (used for trip deletion) does have a focus trap via Radix primitives. The two most user-critical dialogs are not using Radix. Per CLAUDE.md convention and Doppelscript standard, Radix UI should be used for all modals.

**P1. BookingConfirmation focus return on cancel.** `onCancel` sets `showConfirmation(false)` which unmounts the modal. Focus returns to wherever the DOM event last landed, not to the element that triggered the modal (the "Book now" quick reply chip). WCAG 2.1 SC 2.4.3 requires that focus be managed when dialogs close.

**P2. Inline form error `<p>` elements have no `role="alert"`.** In `login/page.tsx:103` and `register/page.tsx:144`, inline validation errors are rendered as `<p className={styles.error}>`. Screen readers do not announce these on injection. `role="alert"` or `aria-live="assertive"` is required.

**P2. Chat form has no `aria-live` announcement of agent state.** When a message is sent and the agent starts working, there is no programmatic announcement. Blind users have no feedback that their message was received.

**P2. Send button text is `"..."` during sending with no accessible label change.** The send button content changes from "Send" to "..." while `isSending`. There is no `aria-label` update. Screen readers will announce the button as containing "..." which is meaningless.

**P2. PreferencesWizard step circles for uncompleted steps are not keyboard-reachable.** `tabIndex={isCompleted ? 0 : -1}` makes uncompleted step circles non-focusable. A keyboard user cannot navigate the progress bar to review which steps remain. This is not a blocker (clicking an uncompleted step does nothing) but it creates a confusing hole in the tab order.

**P2. Weather table in destination detail uses `role="table"` with non-semantic markup.** `explore/[slug]/page.tsx:183-229` constructs a weather table using `<div role="table">`, `<div role="row">`, `<span role="columnheader">`, and `<span role="cell">`. This is correct ARIA usage for a non-native table but requires careful nesting. The table has a `rowheader` for "High", "Low", "Rain" rows but the first row (month headers) has no `rowheader` or `columnheader` relationship via `aria-describedby`. A screen reader will announce column headers without associating them to cells. **P2: convert to a native `<table>` element with `<thead>`, `<tbody>`, `<th scope="col">`, and `<th scope="row">`.**

**P2. `prefers-reduced-motion` global override does not cover SCSS module `@keyframes` with `!important` in individual files.** The global override in `globals.scss` uses `!important` to collapse all animations. However, `BookingConfirmation.module.scss` defines `.spinner { animation: spin 0.8s linear infinite; }` and `.checkmark { animation: scaleIn 0.3s var(--ease-out); }` and `.overlay { animation: fadeIn 0.2s var(--ease-out); }`. The global override will suppress these correctly since it targets `*` with `!important`. Confirmed working. No additional action required here, but note that the `bookingState` / `spinner` / `checkmark` CSS classes are present in the SCSS but not used in the current `BookingConfirmation.tsx` (which no longer has a staged animation flow). These are dead CSS. Low risk.

**Positive a11y findings:**

- Explore page: filter pills have `aria-pressed`. Destination cards have `aria-label` combining city and country. Image fallbacks have `aria-hidden="true"`. All correct.
- Trip cards: delete buttons have `aria-label="Delete trip"`. Correct.
- ChatBox: chat input has `aria-label`. Correct.
- Header: `<nav aria-label="Main navigation">`. Incomplete preferences badge has `aria-label="Preferences incomplete"`. Correct.
- Tile cards: FlightCard, HotelCard, CarRentalCard, ExperienceCard all have `aria-pressed={selected}`. Correct.
- PreferencesWizard: `role="dialog"`, `aria-modal`, `aria-label`, Escape key closes. Step circles have dynamic `aria-label`. Correct (focus trap gap remains).

The sequencing is better than the prior audit found.

- **Terminology:** "Save itinerary" vs "Booked" vs "saved" (DB status). The BookingConfirmation button now says "Save itinerary" (accurate: it is a portfolio demo, nothing is booked). The trip detail page shows a "Booked" badge when `status === 'saved'`. UX-21 (unify "Booked" vs "Saved") is still unresolved. The button and badge use different words for the same action. Low-impact for a portfolio demo.

- **Wizard cognitive load:** Six steps with no "skip all" option. The user must click Skip six times or Next six times. A "Skip all for now" button that closes the wizard in one action would reduce friction for users who want to get to trip planning immediately.

## Consistency and Patterns

- **Modal implementation inconsistency:** Radix AlertDialog is used for trip deletion (correct). PreferencesWizard and BookingConfirmation are hand-rolled overlays with manually implemented Escape handling, overlay click-to-close, and no focus trap. Per project CLAUDE.md: "Radix UI for headless primitives." Per Doppelscript baseline: all modals use Radix Dialog or AlertDialog. This inconsistency creates a11y gaps and maintenance debt.

- **Loading state inconsistency (UX-09):** Three routes have three different loading treatments. Unresolved.

- **Error handling inconsistency:** Chat errors use Toast (correct). Trips list error uses inline `<p className={styles.error}>` (inconsistent). Trip detail error uses a `<p>Trip not found.</p>` (inconsistent). The project CLAUDE.md says "Use Toast component for API/server errors." The trips list and trip detail pages do not follow this convention.

- **`aria-live` missing across status elements:** No `aria-live` regions observed anywhere in the inspected components. Loading states, tool progress labels, empty states, and sending indicators all rely on visual rendering. Screen reader users have no programmatic notification of any dynamic state change.

- **Doppelscript comparison:** Doppelscript has `jest-axe` assertions in component tests. Voyager has no `jest-axe` integration at the component test level. Accessibility violations will not be caught by the test suite; they require a Lighthouse run or manual inspection. This is the structural gap that allowed the Toast `role="alert"` issue to persist without detection.
