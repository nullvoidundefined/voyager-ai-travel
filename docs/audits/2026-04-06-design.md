# Voyager Design Audit

**Date:** 2026-04-06
**Auditor:** Chief Design Officer (canonical role per `~/.claude/audits/design.md`)
**Scope:** `web-client/` (Next.js 15 App Router). Global SCSS, per-page modules, shared components, chat widgets, destination detail pages, trips flow, hero imagery, motion.

---

## Executive Summary

Voyager presents a coherent Mediterranean-warmth brand intent. A single place (`globals.scss`) defines color, radius, shadow, and transition tokens. Those tokens are used with discipline in almost every module. The serif/sans pairing (Fraunces + Outfit) is genuine and aspirational, and the CTA coral against cool stone gives the product a distinctive voice against a sea of blue SaaS travel sites. This is, overall, a more considered visual system than most prototypes.

However, the system stops at color and chrome. There is **no typography scale token** (no `--font-size-*`, `--line-height-*`, `--font-weight-*`), and the type values are scattered across 30+ SCSS modules as raw pixel literals. We counted over 140 unique `font-size: Npx` declarations. The same semantic (a card title, a meta caption, a section header) is executed at 3 to 5 different sizes across pages. This is the single highest-impact drift in the product. Second-highest: `prefers-reduced-motion` is not respected anywhere despite 10+ `@keyframes` animations shipping across Toast, ChatBox, MockChatBox, PreferencesWizard, BookingConfirmation, and spinners. Third: tile components (`HotelCard`, `ExperienceCard`) are near-duplicates with subtle divergence in font sizes, padding, and gap, flagging that the design system lacks a shared "tile" primitive.

**Top 3 priorities:**

1. **P1: Publish a typography scale as CSS custom properties and refactor every module to consume it.** No new font-size literals allowed in SCSS modules.
2. **P1: Add a global `@media (prefers-reduced-motion: reduce)` block that neutralizes every `@keyframes` and transform-based transition.** This is also an a11y compliance issue and will be caught by Lighthouse.
3. **P2: Extract a shared `TileCard` primitive.** `HotelCard`, `ExperienceCard`, and (to a lesser extent) the explore grid card all implement the same pattern with inconsistent dimensions and type.

## Visual Identity & Brand Coherence

**Strengths.**

- The palette in `web-client/src/app/globals.scss` is strong and intentional: warm stone backgrounds (`#faf8f5`, `#f5f2ee`), deep navy-slate foreground (`#1a2332`), Mediterranean blue accent (`#1e56a0`), and a coral CTA (`#e8614d`). This reads like a curated travel brand, not a generic SaaS.
- Fraunces for display and Outfit for body is a confident pairing. The italic `.titleAccent` in `page.module.scss:75` ("planned by AI.") is the single best brand moment in the product.
- The custom compass-style logo mark in `Header.tsx:48` is on-brand and sized modestly. The hover translation (`Header.module.scss:38`) is a nice micro-interaction.

**Gaps.**

- There is no brand-mark asset file. `web-client/public/` contains only `.vibelens/docs/` markdown; there is no favicon, OG image, or logo SVG beyond inline JSX. This hurts social sharing and browser tab identity. **P2.**
- The "Voyager" wordmark lives only as `{APP_NAME}` text. When the page loads before webfonts, it flashes in Georgia fallback, which is acceptable, but there is no `font-display: swap` strategy or preloaded font in `layout.tsx`. **P3.**
- Brand "voice" is carried by the hero copy and CTA color, but is absent everywhere else. Account, trips, and explore list pages read generic. **P2.**

## Typography

**This is the weakest area of the system.** Type is the load-bearing issue in this audit.

**Findings.**

- `globals.scss:4-6` defines only two font-family tokens. There is no type scale: no `--text-xs`, no `--text-display-lg`, nothing.
- `font-size` values are hardcoded pixel literals across every module. Examples of drift:
  - **Page section titles ("h2"):** `page.module.scss:142` `40px`, `explore/[slug]/page.module.scss:107` `28px`, `trips.module.scss:15` `36px`, `faq.module.scss:13` `42px`, `tripDetail.module.scss:128` `24px`. Five different values for the same semantic role. **P1.**
  - **Card titles ("h3"):** `page.module.scss:217` `22px`, `page.module.scss:261` `20px`, `explore/page.module.scss:240` `20px`, `HotelCard.module.scss:60` `13px`, `FlightCard.module.scss:65` `13px`, `ExperienceCard.module.scss:60` `13px`. The "chat tile" title bucket (13px) and the "page card" title bucket (20-22px) are reasonable as two different sizes, but they are never named.
  - **Metadata/caption:** `11px`, `12px`, and `13px` are used interchangeably for essentially the same role (timestamps, subcopy, helper labels). See `ChatBox.module.scss:38` (`11px`), `HotelCard.module.scss:82` (`11px`), `explore/[slug]/page.module.scss:73` (`12px`), `Footer.module.scss:29` (`13px`).
  - **Hero title:** `page.module.scss:67` is `72px`, `explore/page.module.scss:68` is `60px`, `explore/[slug]/page.module.scss:42` is `48px`. These should be three defined steps on a scale, not three guesses.
- `font-weight` is almost always `600` for emphasis, but occasionally drifts to `700` (`explore/[slug]/page.module.scss:43`, `tripDetail.module.scss:207`) and `500` (`page.module.scss:268`). No `--font-weight-*` tokens exist.
- `line-height` is inconsistent: `1.6` in `globals.scss:66`, `1.7` on hero subtitles, `1.55` in experience cards, `1.3` in hotel names. Some values are clearly intentional, but they are not named, so a future change is a manual grep.
- The demo section title `page.module.scss:176` "See it in action" uses the global h2 rule (40px display). The explore hero title (60px) is visually heavier than the landing hero title (72px vs. 60px). Hierarchy between these two surfaces is not coordinated.

**Recommendation.** Publish a scale. Suggested:

```scss
:root {
  /* Type scale, 1.25 modular */
  --text-2xs: 11px;
  --text-xs: 12px;
  --text-sm: 13px;
  --text-base: 14px;
  --text-md: 15px;
  --text-lg: 17px;
  --text-xl: 20px;
  --text-2xl: 24px;
  --text-3xl: 30px;
  --text-4xl: 36px;
  --text-5xl: 48px;
  --text-6xl: 60px;
  --text-7xl: 72px;

  --leading-tight: 1.2;
  --leading-snug: 1.4;
  --leading-normal: 1.6;
  --leading-relaxed: 1.7;

  --font-weight-regular: 400;
  --font-weight-medium: 500;
  --font-weight-semibold: 600;
  --font-weight-bold: 700;
}
```

Then refactor every `font-size: Npx` in SCSS to `var(--text-*)`. This is the P1 for this audit.

## Color System

**Strengths.**

- The palette in `globals.scss:8-33` is complete and semantically named (`--accent`, `--cta`, `--success`, `--danger`, `--warning`, `--highlight`, `--user-bubble`). This is better than most Tailwind-default sites.
- Usage is disciplined. The only raw hex literals outside `globals.scss` are three occurrences: `explore/[slug]/page.module.scss:37` (`#ffffff`) and `trips.module.scss:113,123` (`#fff`). These should be `var(--cta-text)`. **P3.**
- Contrast on the primary palette is strong. `#1a2332` on `#faf8f5` is ~13:1 (AAA).

**Gaps.**

- The CTA color (`#e8614d`) against the Mediterranean blue accent (`#1e56a0`) is visually tight but not always intentional. The "Start Planning" button is coral; the "filterPillActive" in `explore/page.module.scss:158` is also coral; the "Book" button is coral; but the "New Trip" button in `trips.module.scss:24` is also coral. Coral is used for _every_ primary action, which is fine, but there is no secondary action treatment. Result: every page looks like it has one action, which flattens hierarchy on pages with both a "save" and a "new" action. **P2.**
- Focus ring `globals.scss:97` is `2px solid var(--accent)` blue, but the CTA button on hover uses `--shadow-glow` (coral glow). A keyboard user tabbing to a coral button gets a blue focus ring over a coral glow. The combination is visually noisy. Consider `--focus-ring-cta` when the active element is a CTA. **P3.**
- Inline `rgba(30, 86, 160, 0.1)` and `rgba(30, 86, 160, 0.12)` and `rgba(30, 86, 160, 0.15)` appear in `explore/page.module.scss:114,287`, `ChatBox.module.scss:239`, `globals.scss:121`. Same color, three different alphas for essentially the same "faint accent tint" role. Define `--accent-faint`, `--accent-subtle`, `--accent-wash`. **P2.**
- `--accent` and `--accent-text` are identical values (both `#1e56a0`). Keeping two names for one value is either a bug or a future-proofing comment that was never documented. **P3.**

## Layout & Spacing

**Findings.**

- `globals.scss` defines `--radius-sm/md/lg/xl` (8/12/16/24) and three shadow tokens. Good.
- There are **no spacing tokens**. Every margin, gap, and padding is a raw pixel literal: `32px`, `24px`, `20px`, `16px`, `14px`, `12px`, `10px`, `8px`, `6px`, `4px`. The values cluster on an 8-px rhythm most of the time, but `14px`, `18px`, `22px`, `28px` sneak in often enough that the rhythm is broken on close inspection. Example: `ChatBox.module.scss:51` `14px 18px`, `ChatBox.module.scss:213` `18px 24px`. **P2.** Publish `--space-1..12` on an 8-px base (with 4-px for tight cases).
- Max-width is duplicated: `1400px` in `Header.module.scss:13`, `Footer.module.scss:7`, `page.module.scss:2`, `trips.module.scss:3`, `explore/page.module.scss:3`, and `1200px` in `tripDetail.module.scss:3`, and `1000px` in `explore/[slug]/page.module.scss:89`. Should be `--container-wide`, `--container-default`, `--container-reading`. **P2.**
- Negative-margin "break out" trick on `page.module.scss:20` and `explore/page.module.scss:20` is repeated verbatim. Extract to a utility class or composable mixin. **P3.**
- The destination detail page `explore/[slug]/page.module.scss:36` uses `padding: 32px 48px` on the hero but `padding: 0 32px` on the content (`:89`). The left edge of the hero title (48px) does not align with the left edge of body paragraphs (32px + whatever the 1000px container adds). A designer aligning guides in Figma would flag this immediately. **P2.**

## Hero & Imagery

**Findings.**

- Imagery strategy is to load Unsplash CDN URLs via `web-client/src/lib/destinationImage.ts`. There are no local optimized assets. `web-client/public/` has zero real images. This means: (a) every destination image is a third-party dependency, (b) there is no branded fallback imagery, (c) the site has no OG image, no favicon beyond the default Next.js one, and no logo asset. **P2.**
- Alt text is present on every `Image` and `img` (see grep on `alt=`). Good. But the alt text is low quality. `explore/page.tsx:124` `alt={dest.name}` (just "Tokyo") is not descriptive for screen readers. And for the hero carousel, `alt={`${img.city} destination`}` is repeated five times in the DOM; only one of the stacked hero images should have meaningful alt text and the rest should be `alt=""` since they are decorative duplicates of the active one. This is shared territory with the UX/a11y auditor. **P2.**
- Unsplash URLs are called with `w=1600&h=800&q=80`. Reasonable, but the destination card grid at `cardImage` `height: 200px` is loading `w=600&h=300` which is correct for desktop but underspecified for retina. Consider `&dpr=2`. **P3.**
- The hero carousel cross-fade (`page.module.scss:31` `transition: opacity 1s ease-in-out`) is 1000ms linear-ish cross-fade, while other transitions on the same page are 150ms/300ms cubic-bezier from `--ease-out`. The hero fade should be tokenized (`--transition-hero: 1s ease-in-out`) or reuse `--ease-out`. **P3.**
- Hero fallback in `explore/[slug]/page.module.scss:19` is a linear gradient `var(--accent) 0%, var(--cta) 100%`. The trips card fallback in `trips.module.scss:80` is the same gradient. Extract `--gradient-brand`. **P3.**

## Component Design

**Buttons.**

- `page.module.scss:94` `.primaryCta`, `trips.module.scss:20` `.newTrip`, `auth.module.scss` submit button, `ChatBox.module.scss:248` `.sendButton`, `explore/[slug]/page.module.scss:362` `.ctaButton`, and `Header.module.scss:85` `.signIn` all implement the same "coral CTA" but with different padding (14/36, 11/24, 9/22, 12/24, 14/36), different font-size (15/14/14/14/18/14), and different radii (`--radius-sm` vs `--radius-md`). Six buttons, six implementations. **P1: extract `<Button variant="primary" size="md|lg">`.** This is the canonical "component not abstracted" finding.
- Secondary button: `page.module.scss:117` `.secondaryCta` (hero) and `ChatBox.module.scss:187` `.tryAgainButton` are both "ghost" buttons with different borders (`1.5px rgba(255,255,255,0.7)` vs `1.5px var(--border)`). Should share a Button ghost variant.

**Cards.**

- `HotelCard.module.scss` and `ExperienceCard.module.scss` are **structurally identical** (same `.card`, `.imageArea: 120px`, `.body`, etc.) but with drifted type. `ExperienceCard` uses `.cost` at `15px` display font, `HotelCard` uses `.perNight` at `16px` display font. Both claim `width: 200px` but only "HotelCard" uses an `.imageFallback` gradient. **P2: create `components/ui/TileCard.tsx`.**
- Explore destination card (`explore/page.module.scss:190`) is a third tile variant with `border-radius: var(--radius-md)` and `height: 200px` image. Related but different enough to keep separate, though it should consume shared `--tile-*` tokens.
- Trip card in `trips.module.scss:49` is a fourth variant. Different image height (`160px`), different title size (`20px`), same coral fallback gradient.

**Form inputs.**

- Two search inputs: `explore/page.module.scss:90` and `ChatBox.module.scss:218`. Both `1.5px solid var(--border)`, both `border-radius: var(--radius-sm)`, but one is `padding: 12px 20px` and the other is `12px 18px`, and they have different focus states. The focus box-shadow `0 0 0 3px rgba(30,86,160,0.1)` is duplicated verbatim. Extract an input primitive. **P2.**

**Dialogs/modals.**

- `BookingConfirmation` and `PreferencesWizard` are the two modal-style surfaces and they both implement their own overlay + slide-up animation. No shared `Dialog` primitive. Voyager's stack allows Radix UI per project conventions, but there is no `components/ui/` directory at all. **P1 (convention violation): the project convention file mandates Radix primitives styled with SCSS modules, and a `components/ui/` folder for extracted shared patterns. Neither exists.**

## Iconography

- Icons are **all inline SVG** inside `.tsx` files (see `page.tsx:16-94` for 4 feature icons, `Header.tsx:48-106` for the logo compass). No icon library, no `components/ui/icons/` module, no Lucide/Heroicons dependency. **P2: extract to a shared icon set.** Stroke widths drift: the feature icons use `1.5`, the logo uses mixed `1.5`, and the chevron in `ItineraryTimeline.module.scss:58` is a text glyph not an SVG. Icon strokes should all be `1.5` or all be `2`, not mixed.
- There are no empty-state illustrations. `trips.module.scss:179` `.empty` is a text block. Explore filtered-to-zero shows "0 destinations" count. These are opportunities for branded empty state art. **P3.**

## Motion & Animation

**Findings.**

- Animation tokens in `globals.scss:47-49`: `--ease-out: cubic-bezier(0.16, 1, 0.3, 1)`, `--transition-fast: 0.15s`, `--transition-med: 0.3s`. Good foundation.
- `@keyframes` count: **11 unique keyframe definitions** across 8 files (`fadeIn`, `slideUp`, `spin`, `scaleIn`, `fadeSlideIn` duplicated in 3 files, `typingDot` duplicated in 2 files). The duplication is P2 drift: `ChatBox.module.scss:275` and `VirtualizedChat.module.scss:85` and `MockChatBox.module.scss:136` all define `@keyframes fadeSlideIn` separately, and there is no guarantee they stay in sync. Hoist to `globals.scss`.
- **`prefers-reduced-motion` is not referenced anywhere.** Zero files match. This is a hard a11y violation: the product has 10+ keyframe animations and multiple `transform: translateY(-2px)` hover states but does nothing when the user sets reduced-motion at the OS level. **P1.** Fix with a global:

```scss
@media (prefers-reduced-motion: reduce) {
  *,
  *::before,
  *::after {
    animation-duration: 0.01ms !important;
    animation-iteration-count: 1 !important;
    transition-duration: 0.01ms !important;
    scroll-behavior: auto !important;
  }
}
```

- Hero cross-fade duration (1s) does not match any defined transition token. **P3.**
- Spinner keyframes `@keyframes spin` appear in three separate files (`ToolProgressIndicator.module.scss:38`, `BookingConfirmation.module.scss:273`, `newTrip.module.scss:44`). Hoist. **P2.**
- Hover transforms `translateY(-1px)` and `translateY(-2px)` and `translateY(-3px)` are used for the "lift on hover" pattern on buttons, feature cards, and destination cards respectively. Three different lift distances, no token. **P3.**

## Design System Maturity

**Maturity level: 2 of 5.** ("Tokens defined for color/radius/shadow; nothing else; components not abstracted.")

- Tokens that exist: color (14), radius (4), shadow (4), transition (3), fonts (2). **Total ~27.**
- Tokens that should exist and don't: type scale (12+), spacing (8+), line-height (4), font-weight (4), z-index (3-4), container widths (3), gradient brand (1), motion curves (2+). **Gap ~35+.**
- `components/ui/` directory: missing. Every shared-pattern candidate (Button, Input, TileCard, Dialog, Badge) is inlined per-file.
- Documentation: none. No Storybook, no MDX, no inline component stories.
- Radix UI: **not installed** per project convention. The convention file `.claude/bottomlessmargaritas/CLAUDE-FRONTEND.md` (referenced from CLAUDE.md) requires Radix for interactive primitives (dialogs, dropdowns, toggles). `PreferencesWizard` and `BookingConfirmation` are custom-rolled instead. **P1 (convention violation).**
- Prettier/lint catch some drift, but the real design-drift problem (type scale, spacing scale) is invisible to linting. Consider a stylelint rule banning raw `font-size` and `margin` pixel values in `.module.scss` files once the tokens land.

## Visual Hierarchy & Scannability

- Landing page hierarchy is good. Hero → demo → how-it-works → features → explore → final CTA is a clear narrative. The section titles at 40px and subtitles at 17px create a clear rhythm.
- Destination detail page is **information-dense without enough breathing room.** Quick Stats bar → About → Top Experiences (2-col grid) → Dining → Neighborhoods → Weather (12-col grid) → Advisories → CTA. The `padding: 40px 0` between sections (`explore/[slug]/page.module.scss:97`) is the only visual separator beyond a hairline border. For a destination like "Tokyo" with dense content, the eye gets tired. Consider alternating background (`var(--surface)` vs `var(--surface-alt)`) on every other section. **P2.**
- The weather 12-column grid at `weatherRow: 60px repeat(12, 1fr)` with `min-width: 700px` is unreadable on mobile without horizontal scroll, and the horizontal scroll is not visually indicated (no gradient mask, no scroll hint). **P2.**
- Trip detail page mixes a destination hero with a full-width header below it, then a budget card, then a cost breakdown, then an itinerary. The `--destinationHeroOverlay` dark gradient + 42px white title is the strongest visual element on the page and creates a "second fold" anchor that is good. But the "Cost Breakdown" section title is `24px` (`tripDetail.module.scss:129`) while the page `h1` is `42px`. The gap is too large. Intermediate section titles should be ~30-32px. **P3.**

## Responsive Design

Tested breakpoints: 375, 390, 414, 768, 1024, 1440 (logical review of modules).

**375-414 (mobile):**

- Landing hero `44px` title (`page.module.scss:354`) is fine.
- Explore grid collapses to 1 column at `600px` (`explore/page.module.scss:332`). Good.
- Chat box height is hardcoded at `620px` (`ChatBox.module.scss:8`). On a 375x812 iPhone with a 64px header and 24px padding, 620px forces scroll and the input area pushes below the fold. **P2: use `height: min(620px, calc(100vh - 160px))` or similar.**
- Weather chart forces horizontal scroll on `min-width: 700px`. As noted above.
- Filter pills on `explore` page: at `800px` breakpoint they become `flex-wrap: nowrap` (`explore/page.module.scss:328`) with `overflow-x: auto`, but there is no visual affordance (fade, chevron, scroll indicator) that more pills exist off-screen. **P2.**
- Hero carousel at 375: the 5 stacked `<Image>` elements with `fill` and `priority={i === 0}` is correct, but the priority hint is only on the first, meaning the carousel auto-advance at 5s intervals forces late loads that cause a flash on slow connections. **P3.**
- Mobile nav: `Header.module.scss:108` has no hamburger. At 600px the nav links shrink to `13px` and share a row with the logo and sign-in button. At 4 links (Explore, Trips, Account, FAQ) plus logo plus sign-in, this is tight on 375. **P2: add a hamburger at ≤ 640px.**

**768-1024 (tablet):**

- Explore grid at 2 columns via `(max-width: 900px)` (`explore/page.module.scss:291`). Fine.
- Feature grid on landing goes from 4 → 2 at `800px`. Fine.
- Destination detail experience grid 2 → 1 at `700px`. Fine.

**1440 (desktop):**

- `.landing` caps at `1400px`. Hero breaks out to `1600px`. The transition between capped content and full-bleed hero is smooth but the 1600px cap means on a 1920px monitor the hero has 160px of empty gutter on each side. Decide: full-bleed or capped. Capping at 1600 but filling to 1920 would be cleaner. **P3.**

**Touch targets.** The chevron in `ItineraryTimeline.module.scss:56` is a text glyph inside a 40px button row. Tap area is OK. The delete button on trip cards (`trips.module.scss:108`) is 32x32. Below WCAG recommended 44x44. **P2.**

## Polish & Craft

**Strong:**

- Hover states exist almost everywhere (buttons, cards, nav, inputs).
- Focus-visible is globally defined (`globals.scss:96`).
- Input focus ring is a thoughtful 3px blue glow (`explore/page.module.scss:114`).
- Selection color `globals.scss:121` is branded.
- Custom scrollbar `globals.scss:101-117` is restrained (6px, no track).

**Weak:**

- **Skeleton screens: none.** `trips.module.scss:190` `.loading` is a centered text block "Loading…". Destination detail page shows nothing during fetch. Chat messages during tool calls use a typing indicator (good) but the overall trip list and destination list show bare text. **P2.**
- **Active states on buttons: missing almost everywhere.** Only `explore/[slug]/page.module.scss:383` has `&:active { transform: translateY(0) }`. Other CTAs jump from hover to release without a pressed state. **P3.**
- **Disabled state drift.** `ChatBox.module.scss:243` input disabled is `opacity: 0.5`, `ChatBox.module.scss:270` send button disabled is `opacity: 0.4`. Close, not identical. Define `--disabled-opacity: 0.5`. **P3.**
- **Toast** component (`Toast.module.scss`) is well-built with slideUp animation but uses custom roll instead of Radix Toast, and has no queue/stack handling in the CSS. **P2 (convention violation, shared with Dialog finding).**
- **Error boundary fallback** (`ErrorBoundary.module.scss`) was not inspected in depth, but the fact that it lives as a bespoke module rather than reusing a card primitive is itself evidence of missing abstractions. **P3.**

## Disposition

The Voyager team has a real design sense. The palette, the Fraunces/Outfit pairing, and the coral CTA are stronger than 80% of AI-generated travel products. The problem is the bottom half of the design system is not built: typography, spacing, and component primitives. Those gaps are cheap to fix now and expensive later. A single afternoon of work can publish the type and spacing scales, and a second afternoon can install Radix and extract Button + TileCard. After that, the system will resist drift on its own. Do this before shipping to more users. Every inconsistent font-size the team ships from here forward is a small withdrawal from brand equity.
