# Voyager Design Audit

**Date:** 2026-05-27
**Auditor:** Chief Design Officer (canonical role per `~/.claude/audits/on-request/design.md`)
**Scope:** `web-client/` (Next.js 15 App Router). All SCSS modules, globals, shared components, chat widgets, destination detail, trips flow, explore pages, auth pages, hero imagery, motion.
**Reference standard:** Doppelscript (`packages/tokens/`, Storybook, SCSS modules with token-based spacing/color/typography).
**Prior audit:** `docs/audits/2026-04-06-design.md`. Prior P1 findings DES-01 through DES-04 are referenced below.
**Known existing issues:** DES-05 through DES-10 per ISSUES.md. Not re-reported.

---

## Executive Summary

Seven weeks on from the April audit, three prior P1 findings have partial remediation. DES-01 (type scale) added the token definitions to `globals.scss` but 253 raw `font-size: Npx` literals remain across 37 SCSS modules, with only 3 declarations actually consuming those tokens -- all in a single component (`AlertDialog`). The token scaffold exists; the migration has not happened. DES-03 (prefers-reduced-motion) is correctly handled by the global block in `globals.scss`. DES-04 (components/ui/) produced exactly one entry: `AlertDialog`. The design system is still a single-room house.

The three new highest-priority findings from this audit are:

1. **DES-11 (P1): The `DemoBanner` component uses a completely foreign token vocabulary** -- `--color-bg-soft`, `--color-border`, `--color-accent-dark` -- none of which exist in `globals.scss`. These values fall back to hardcoded hex constants. The banner currently appears on most pre-auth pages and is the first visual element users see after the header.

2. **DES-12 (P1): `AlertDialog` maps `--accent` to `#c2410c` (a burnt orange) as its confirm button background**, which is not the Mediterranean blue (`#1e56a0`) that `--accent` resolves to in `globals.scss`. The component either has a stale token copy or was designed against a different system. Every destructive action in the product (delete trip) routes through this dialog.

3. **DES-13 (P2): No spacing token system exists.** Every padding and gap value is a raw pixel literal. The product has 8, 10, 12, 14, 16, 18, 20, 24, 28, 32, 36, 40, 44, 48 -- at least 14 distinct spacing values -- none of which are named. Doppelscript uses a `spacing` token scale; Voyager does not. Below Doppelscript quality bar.

Brand coherence is functional but not polished. The visual voice of the landing page hero (Fraunces, coral CTA, warm stone palette) does not carry into the inner application. Trips list, account, and trip detail pages are visually generic. The product has no favicon, no OG image, and no logo SVG file -- all three are missing from `public/`. This is a ship-stopping brand gap for a product with a production domain.

## Visual Identity & Brand Coherence

**Strengths (unchanged from April).**

- The Mediterranean-warmth palette remains distinctive and intentional. Warm stone backgrounds, deep navy-slate foreground, Mediterranean blue accent, and coral CTA. This is not a generic SaaS palette.
- The Fraunces + Outfit pairing continues to be the product's strongest visual differentiator. The italic `.titleAccent` on the landing page is the best brand moment in the product.
- Token discipline on color and radius is solid. Essentially every module uses `var(--surface)`, `var(--border)`, `var(--accent)` etc. No raw hex color drift in the core palette (the exceptions are catalogued below in the color section).

**Gaps.**

- **No public brand assets.** `web-client/public/` contains only VibeLens documentation markdown. No `favicon.ico`, no `apple-touch-icon.png`, no OG image, no logo SVG. On production at `interviewiangreenough.xyz`, the browser tab shows a blank favicon and social shares produce a link preview with no image. For a product in production this is a P1 gap. **DES-11-adjacent, new finding.**
- **Brand voice drops at the auth boundary.** The landing and explore pages carry the aspirational voice ("Plan the trip of a lifetime"). Auth, account, trips list, and trip detail pages read like generic admin UI. The system has the palette and typography to carry brand warmth everywhere; it is not being applied. P3.
- **Logo color inconsistency.** In `Header.module.scss`, the wordmark and icon are colored `var(--cta)` (coral, `#e8614d`). In `Footer.module.scss`, the wordmark is colored `var(--foreground)` (navy-slate, `#1a2332`). The logo changes color between top and bottom of every page. This is style drift, not an intentional distinction. **P2.**

## Typography

**Status: DES-01 token definitions landed; migration adoption is near-zero.**

The April audit flagged 140+ raw `font-size` literals. The fix was to publish token definitions in `globals.scss` (DES-01, done). Token adoption has not followed. Current count: **253 raw `font-size: Npx` declarations** across 37 SCSS modules. Token usage: **3 declarations**, all in `AlertDialog.module.scss`, all with pixel fallbacks that undercut the token's purpose.

**Specific drift (new, post-April):**

- **Section title h2 drift is unresolved.** Values observed across current modules: `42px` (faq.module.scss), `40px` (page.module.scss), `36px` (trips.module.scss, account.module.scss), `28px` (explore/[slug]/page.module.scss), `24px` (tripDetail.module.scss -- breakdown h2, itinerary h2, chat section h2). A `--text-3xl: 36px` token exists. It is not being used for any of these.
- **Hero title scale is inconsistent across parallel surfaces.** Landing page h1: `72px`. Explore list page hero h1: `60px`. Explore slug hero title: `48px`. These three pages have structurally identical hero layouts (image + overlay + title + subtitle). The title sizes differ without a scale to justify the difference. All three values are in the `globals.scss` token set (`--text-4xl: 48px`, `--text-5xl: 64px`, `--text-6xl`... wait -- the token set tops at `--text-5xl: 64px` and the landing uses `72px`, which is not a defined token). The landing hero title is off the top of the published scale.
- **Chat widget font sizes.** HotelCard name: `13px`. ExperienceCard name: `13px`. FlightCard airline: `13px`. These match. CarRentalCard provider: also `13px`. But `WeatherForecast` day card date: `11px`, condition: `11px`, temperature high uses `--font-display` at `16px`, low at `13px`. The temperature label alone introduces three sub-token sizes into a single small card.
- **`font-weight: 700` drift.** The brand has a 600-weight system. Seven locations use `700`: `faq.module.scss:33` (section h2 label), `explore/[slug]/page.module.scss:43` (hero title), `account.module.scss:46` (avatar letter), `tripDetail.module.scss:156` (cost row remaining), `tripDetail.module.scss:209` (day number), `CarRentalCard.module.scss:58` (logo fallback), `AdvisoryCard.module.scss:59` (severity label), `FlightCard.module.scss:51` (logo fallback), `BookingConfirmation.module.scss:246` (checkmark). No `--font-weight-bold` token exists, so these are invisible to design-system enforcement.
- **`ErrorBoundary.module.scss` uses `rem` and `em` units.** Every other component in the codebase uses `px`. The ErrorBoundary uses `2rem` for padding, `1.5rem` for heading, `0.875rem` for button. This is not a functional issue but it is a consistency failure: a developer reading the codebase will be confused about which unit system to use. **P3.**

**Compared to Doppelscript quality bar:** Doppelscript token package generates both JS tokens and SCSS variables from a single source of truth, then enforces token-only usage in modules via lint. Voyager has the token definitions but no enforcement, no adoption, and the tokens themselves do not cover the full range of sizes in use (the `72px` landing hero is off-scale).

## Color System

**Status: Core palette is solid. Two P1 violations in secondary components.**

**Finding DES-12 (P1): `AlertDialog.module.scss` uses a wrong accent value.**
The `confirmButton` in `AlertDialog.module.scss` sets `background: var(--accent, #c2410c)` with a fallback of `#c2410c` (burnt orange, clearly from a different design system). The `--accent` token resolves to `#1e56a0` (Mediterranean blue) in `globals.scss`. The fallback is never triggered in the running app -- `--accent` always resolves -- but the comment in the fallback reveals that the component was originally authored against a different token system. More concerning: the `confirmButton` style is semantically wrong; a "confirm" action in Voyager's design language should use `--cta` (coral) not `--accent` (blue). The destructive button correctly uses `#dc2626` directly, but the confirm path is broken.

**Finding DES-11 (P1): `DemoBanner.module.scss` uses entirely foreign tokens.**
`--color-bg-soft`, `--color-border`, `--color-text-muted`, `--color-accent-soft`, `--color-accent-dark`, `--color-accent` -- none of these are defined in `globals.scss`. The fallback hex values are orange (`#fff7ed`, `#fed7aa`, `#7c2d12`, `#c2410c`) -- this is an Orange/Tailwind palette that is completely orthogonal to the Mediterranean-warmth system. The banner's amber/orange warning treatment has no relationship to `--warning` or `--cta`. Users see a design that looks like it belongs to a different product.

**Other color findings (P2/P3):**

- Scattered `rgba(30, 86, 160, ...)` literals across 7 modules (faq, explore, auth x3, TripDetailsForm, ChatBox). This is the accent blue deconstructed. A `--accent-ring` or `--accent-focus` token is needed to consolidate these focus-ring shadows.
- `rgba(22, 163, 74, 0.08)` appears in both `trips.module.scss` and `tripDetail.module.scss` for the "saved/booked" status badge background. A `--success-light` token does not exist; this calculation is duplicated.
- `rgba(220, 38, 38, 0.85)` for the delete button hover in `trips.module.scss` is an un-tokenized danger value.
- `explore/[slug]/page.module.scss:37` uses `color: #ffffff` inline rather than `var(--cta-text)` which resolves to `#ffffff`. Functionally identical but not using the token.
- The `Toast` component is hard-coded to `background: var(--danger)`. It has no success or info variant. Every toast is a red error toast, including potential future success notifications. **P3.**

## Layout & Spacing

**No spacing tokens exist. This is the most systemic gap relative to Doppelscript.**

The Doppelscript reference standard uses a `spacing` token scale (4px base grid). Voyager uses raw px literals everywhere. The following distinct spacing values appear in padding/gap declarations across the modules reviewed:

4, 5, 6, 8, 10, 11, 12, 13, 14, 16, 17, 18, 19, 20, 22, 24, 26, 28, 32, 36, 40, 44, 48, 52, 56, 60, 64, 72, 80, 100

That is over 30 distinct spacing values with no naming, no scale rationale, and no enforcement. The lack of a spacing system is the single largest structural gap versus Doppelscript. Below Doppelscript quality bar.

**Page container width inconsistency.** Content containers use: `1400px` (landing, explore list, footer/header), `1200px` (trips, trip detail, new trip), `1060px` (faq), `1000px` (explore slug content), `860px` (account). Five different max-widths. Doppelscript uses a single container width token. Voyager needs to decide whether the constraint is `1200px` (inner app) and `1400px` (marketing pages) and document the decision as tokens.

**Breakpoint inconsistency.** Breakpoints observed: `900px`, `800px`, `700px`, `600px`, `560px`, `480px`. No breakpoint token system exists. Doppelscript uses named breakpoints (`sm`, `md`, `lg`, `xl`). Voyager's breakpoints are typed from memory in each module, resulting in the current six-variant set with no semantic alignment. A mobile breakpoint of `600px` in one file and `560px` in another will produce different behavior at 580px viewport width. **P2.**

**Grid gap inconsistency.** The explore list card grid uses `gap: 24px` at desktop, `16px` at tablet, while the trips list uses `gap: 24px` at all sizes (collapses to single column but keeps the gap). The how-it-works section uses `gap: 32px`. Features section uses `gap: 24px`. No gap token exists. **P3.**

## Hero & Imagery

**Hero structure is correct; execution has drift between pages.**

Both landing and explore-list heroes share an identical pattern: `position: absolute` image, `z-index` stacked overlay + content, crossfade animation between images. The implementation is duplicated rather than shared, with subtle value differences:

- Landing hero overlay gradient: `rgba(0,0,0,0.3)` to `rgba(0,0,0,0.55)`
- Explore list hero overlay gradient: `rgba(0,0,0,0.3)` to `rgba(0,0,0,0.6)`
- Explore slug hero overlay: `rgba(0,0,0,0.15)` to `rgba(0,0,0,0.6)` (direction reversed: `to top`)
- Trip detail destination hero: `transparent 40%` to `rgba(0,0,0,0.6)` (also bottom-heavy, but no top fade)

These are four hero overlays built from scratch, each slightly different. The explore slug and trip detail overlays are directionally inconsistent with the landing and explore list. A shared `.hero-overlay` mixin or CSS custom property would enforce consistency.

The `heroImage` crossfade animation (`opacity: 0; transition: opacity 1s ease-in-out`) uses `ease-in-out` rather than `var(--ease-out)`. This is the one motion case not covered by the global prefers-reduced-motion block's `transition-duration: 0.01ms` override -- wait, actually it is covered, because the override is `*::before, *::after` plus `*`, which catches this class. The prefers-reduced-motion implementation is correct.

**Alt text coverage.** All `<Image>` and `<img>` elements have non-empty alt text. The existing DES-10 finding (inconsistent decorative vs. meaningful strategy) remains open but there are no new missing-alt violations to report from this audit sweep.

**Missing brand assets.** No favicon, no OG image. These are ship-stopping for a production product on a custom domain. Every social share of `interviewiangreenough.xyz` produces a bare link with no image preview.

## Component Design

**Buttons: three inconsistent confirmation styles.**

The confirm action uses three different visual treatments depending on context:

1. `SelectableCardGroup.confirmButton`: `background: var(--accent)` (blue)
2. `TripDetailsForm.submit`: `background: var(--accent)` (blue)
3. `BookingConfirmation.confirmButton`: `background: var(--cta)` (coral)
4. `AlertDialog.confirmButton`: `background: var(--accent, #c2410c)` (blue in practice, broken fallback)
5. Auth page submit: `background: var(--cta)` (coral)
6. Trips list "New Trip" button: `background: var(--cta)` (coral)

The primary action button in the chat flow (TripDetailsForm) uses blue (accent). The primary action button on the auth page uses coral (CTA). These are the two highest-traffic actions in the product. They look like they belong to different products. The decision of which color is "primary action" is not made at the token level.

**Chip/pill pattern: two implementations with subtle drift.**

`auth.module.scss` `.chip` uses `border: 1.5px solid` and `.chipSelected` fills with `var(--accent)` (blue).
`PreferencesWizard.module.scss` `.chip` uses `border: 2px solid` and `.chipSelected` fills with `var(--cta)` (coral).

Same chip component semantics (multi-select preference toggle), different border widths, different selection colors. This indicates the PreferencesWizard was designed independently from the auth step flow, despite both being part of the registration sequence.

**Card pattern: HotelCard and ExperienceCard are near-identical with value drift.**

Both are 200px wide, both have 120px image areas, both use the same `border: 1.5px solid var(--border)`, `border-radius: var(--radius-md)`, same hover states. The card body padding is `12px` in both. But:

- HotelCard `.name` is `font-size: 13px`. ExperienceCard `.name` is also `13px`. These match.
- HotelCard `.stars` uses `var(--highlight)` (amber `#f5a623`). ExperienceCard `.rating` also uses `var(--highlight)`. These match.
- HotelCard price is 16px via `--font-display`. ExperienceCard cost is 15px via `--font-display`. One pixel of drift on the price label.
- HotelCard has `.dates` at `11px` and `.total` at `11px`. ExperienceCard has no date/total row.
- CarRentalCard is 220px wide (vs 200px for the others), different card structure.

The TileLayout extraction (recent change noted in the brief) correctly unified the scroll container layout. The cards themselves still diverge. The right next step is a shared `.TileCard` SCSS partial that both import, as recommended in the April audit. DES-06 (components/ui/ not established) remains open; only AlertDialog has been added.

**Modal pattern: two implementations.**

`BookingConfirmation.module.scss` and `PreferencesWizard.module.scss` both implement modal overlays with backdrop blur, slide-up animation, and a centered card. They use the same structural approach but:

- `BookingConfirmation` overlay backdrop: `rgba(0,0,0,0.4)` with `blur(8px)`.
- `PreferencesWizard` overlay backdrop: `rgba(26,35,50,0.45)` with `blur(4px)`.
- `AlertDialog` overlay backdrop: `rgba(26,35,50,0.65)` without blur.

Three overlays, three different backdrops. The blur level and opacity communicate different modal "weights." There is no design intent here -- these were built independently. A `--overlay-bg` token and a `--overlay-blur` token would unify them.

## Iconography

Voyager uses emoji and Unicode characters as icons throughout the product (the typing indicator dots are CSS-animated `<span>` elements; the chevron in `ItineraryTimeline` is a plain `+` character that rotates; the FAQ accordion uses `+` that rotates 45deg; the `checkmark` in BookingConfirmation is a `✓` character inside a green circle). This approach is consistent -- the product does not appear to use any icon library. The risk is that emoji rendering varies by OS; the approach is acceptable for a prototype but would need to be formalized (via a proper icon library like Lucide or Phosphor) before the product reaches a broader audience. No violation to file at current scale.

The compass/airplane emoji in the logo hover (`Header.tsx`) is a nice micro-interaction.

## Motion & Animation

**Status: Global prefers-reduced-motion guard is now in place (DES-03, resolved). Easing token usage has gaps.**

The `@media (prefers-reduced-motion: reduce)` block in `globals.scss` correctly covers all elements including pseudo-elements with `!important`. This is the correct approach and DES-03 is resolved.

**New finding: `AlertDialog.module.scss` uses raw easing values, not tokens.**

`animation: fadeIn 150ms ease-out` and `animation: slideUp 180ms ease-out` and `transition: background 120ms ease-out`. The `--ease-out` token and `--transition-fast` token exist. AlertDialog was likely ported from elsewhere without adapting to Voyager's token system.

**Duplicate `@keyframes` across modules.** The `slideUp` animation is defined in: `BookingConfirmation.module.scss`, `PreferencesWizard.module.scss`, and `AlertDialog.module.scss`. The `fadeSlideIn` animation is defined in: `ChatBox.module.scss`, `VirtualizedChat.module.scss`, and `MockChatBox.module.scss`. The `typingDot` animation is defined in: `ChatBox.module.scss` and `MockChatBox.module.scss`. The `spin` animation is defined in: `BookingConfirmation.module.scss` and `newTrip.module.scss`. A global `@keyframes` declaration file or mixin system would eliminate this duplication and ensure consistent timing.

**`ChatProgressBar.module.scss` uses `animation: indeterminate-slide 1.4s var(--ease-out) infinite`.** This is one of the few animations that uses the easing token. The `ease-out` cubic bezier is correct for an indeterminate progress animation -- it reads as deliberate motion.

**No page transition animations.** Navigating between routes is an instant cut. This is appropriate for a data-heavy application but worth noting: the landing page has significant motion investment (hero crossfade, hover transforms, typing indicator) while the inner app has almost none. The mismatch in motion density reinforces the "two different products" perception.

## Design System Maturity

**Below Doppelscript quality bar.**

Doppelscript reference: tokens package with JS + SCSS generation from a single source, Storybook for every component, enforced token-only usage.

Voyager current state:

| Dimension         | Doppelscript               | Voyager                                                           |
| ----------------- | -------------------------- | ----------------------------------------------------------------- |
| Color tokens      | Generated, typed, enforced | Defined in `globals.scss`, used well                              |
| Typography tokens | Generated, typed, enforced | Defined in `globals.scss`, 1% adoption                            |
| Spacing tokens    | Generated, 4px grid        | Not defined                                                       |
| Radius tokens     | Yes                        | Yes, used well                                                    |
| Shadow tokens     | Yes                        | Yes, used well                                                    |
| Transition tokens | Yes                        | Partial (easing token exists, raw overrides in AlertDialog, auth) |
| Breakpoint tokens | Named (sm/md/lg/xl)        | Not defined, 6 inconsistent values                                |
| Component library | Storybook                  | One component in `components/ui/`                                 |
| Documentation     | Storybook stories          | None                                                              |

The token definitions that were added in response to the April audit are the right foundation. The system is stalled at "definitions exist" rather than "definitions are enforced and adopted." The gap between Doppelscript and Voyager is not conceptual -- it is the migration work and the enforcement tooling.

## Visual Hierarchy & Scannability

The landing page hierarchy is effective: hero draws the eye, the demo ChatBox anchors the center column, and the three-step "how it works" section reads cleanly. The section dividers (1px `var(--border-light)`) are subtle enough not to interrupt flow.

The trips list page has a hierarchy problem: the trip card images are 160px tall and dominate the card, but the date and budget information (the things users actually care about) are in small text below. The most useful information is the least visually prominent.

The trip detail page stacks: destination hero -> back link -> header (trip name + budget) -> cost breakdown -> itinerary -> chat. The chat section is at the very bottom of the page. For a product whose core value proposition is the conversational AI loop, the chat input being below all itinerary content means users who have a long itinerary must scroll past it to continue the conversation. This is a UX issue but it manifests as a visual hierarchy problem.

The FAQ page uses `font-size: 12px; text-transform: uppercase` for section headings, which is a deliberate small-caps treatment that reads well as a wayfinding device. However the color is `var(--accent)` (blue), while the account page uses the identical typographic treatment (`font-size: 12px; font-family: var(--font-body); font-weight: 600; text-transform: uppercase`) but colors it `var(--foreground-muted)` (gray). Same visual role, different colors.

## Responsive Design

**Breakpoints audited: 375, 600, 800, 1024, 1400.**

The product uses `max-width: 800px` and `max-width: 600px` as its primary responsive gates. There is no `1024px` breakpoint. At 1024px, the layout uses the desktop grid. At 375px (iPhone SE), the layout uses the sub-600px rules.

**375px (mobile).**

- Landing page: single-column layout, hero title drops to `36px` (appropriate). CTA buttons stack vertically (good).
- Feature grid: single column.
- Auth card: padding reduces to `36px 24px`. Acceptable.
- Header: padding reduces to `0 16px`, logo to `22px`. The nav links and sign-in button are still present side-by-side at narrow widths -- this could become cramped at 375px if the nav has more than 2 links.
- Chat tile cards (200px wide, horizontal scroll): functional at 375px but the user is scrolling a narrow viewport horizontally inside an already-narrow chat message. The horizontal scroll container has no visual affordance indicating scrollability.

**768px (tablet).**

- No `768px` breakpoint exists. The product jumps from `600px` rules to `800px` rules. At 768px, the layout uses `800px`-and-above styles, meaning the explore grid is still 3 columns -- which is workable but tight.
- The explore card images at 768px are 200px tall in a 3-column grid where each column is roughly 240px. The aspect ratio holds.

**1024px (laptop).**

- Full desktop layout. No issues.

**1400px (wide desktop).**

- Content is correctly capped at `max-width: 1400px`. The `1600px` cap on hero sections allows the hero to feel more expansive without the content overflowing.

**Touch target assessment.** The smallest interactive targets are the nav links at `8px 12px` padding (mobile) and the step dots in auth (`10px` circles). The nav link touch target is roughly `32px` tall, which meets the 32px minimum but falls below the recommended 44px. The step dots at `10px` are inaccessible as touch targets (they are visual-only in the current implementation, so this does not create a functional barrier, but would if they became interactive). The chip/pill buttons (`8px 16px` padding) are approximately `34px` tall -- technically accessible but marginal.

## Polish & Craft

**Hover/focus/active states.**

- The global `:focus-visible` rule in `globals.scss` applies `outline: 2px solid var(--accent)` globally. This is the correct approach. Most interactive elements do not override this, which is good.
- However, `PreferencesWizard.module.scss` `.select` overrides focus with `outline: none; border-color: var(--cta)`. This removes the standard focus ring on the select element, replacing it with a border color change. For keyboard users, the border-color change on a select inside a form is less visible than the standard outline, especially when the border is only 1px. DES-09 (focus states inconsistent) is still open -- this is an example.
- `auth.module.scss` form inputs use `outline: none` and replace focus with a `box-shadow: 0 0 0 3px rgba(30, 86, 160, 0.1)`. This is better than removing focus entirely, but the semi-transparent shadow on a light background provides less than 3:1 contrast against the background. This may fail WCAG 2.1 SC 1.4.11 (Non-text Contrast).

**Skeleton screens.** None exist anywhere in the product. All loading states are either a plain text string (`"Loading trips..."`, `"Loading trip..."`) or a CSS spinner. The trips list, trip detail, and account pages all have loading states. The UX audit noted this (UX-09). The design system has no skeleton pattern to extract from. DES-08 (skeleton screens missing) remains open.

**Edge cases.**

- `trips.module.scss` `.cardImageFallback` uses `background: linear-gradient(135deg, var(--accent), var(--cta))`. This is the blue-to-coral gradient. It is visually attractive as a fallback. However, with text overlaid (the trip destination name is rendered over the image via `tripDetail.module.scss`), the contrast ratio of white text on the mid-gradient color should be verified.
- The empty state on the trips list page is a plain paragraph with no illustration, icon, or visual affordance. The text ("You don't have any trips yet.") followed by a CTA button is functional but visually generic. DES-07 (empty state visuals inconsistent) remains open.
- The `MarkdownText.module.scss` `code` inline element uses `font-family: monospace` directly. The product has no monospace font defined in tokens. For a travel AI that may output code (API examples, trip codes), a named monospace token would be appropriate. P3.
