## Executive Summary

**Top 3 priorities:**

1. The budget bar renders broken layout when allocated cost exceeds the total budget -- segments can sum past 100% with no visual cap or warning. A user who is $500 over budget sees a broken progress bar with no indication they have overrun.
2. The reduced-motion fix (`115fcc3`) directly contradicts the project rule and the global `prefers-reduced-motion` reset: it slows the spinner animation on `reduce` to 1.5s instead of stopping it, which is the wrong behavior for users with vestibular disorders.
3. Instant card selection (31324d2) removed the confirmation step from a consequential action (persisting a flight/hotel choice) without surfacing any undo path. The user clicks a tile, the choice fires immediately, and there is no way to change their mind short of asking the agent conversationally.

**Can a new user complete the primary happy path without help?** Yes, but with material friction at the budget overrun case, the "Flexible" toggle discoverability gap, and the irreversible instant-select pattern on tile cards. The core chat-to-booking flow is reachable.

## Findings

### F-01 -- P2 -- Reduced-motion fix uses wrong behavior

**Commit:** 115fcc3
**File:** `apps/client/web/src/app/(protected)/trips/new/newTrip.module.scss:50-55`
**User-visible symptom:** A user who has enabled "Reduce Motion" in their OS (typically to prevent nausea or disorientation) still sees the spinner rotating -- just slightly more slowly. The animation is not stopped.

The fix sets `animation-duration: 1.5s` and `animation-iteration-count: infinite` inside the `prefers-reduced-motion: reduce` block. This is the opposite of the project rule: "use `animation: none`, not shortened duration." It also fights the global reset in `globals.scss` using `!important`, which is why the engineer added `!important` to their local overrides too -- the two `!important` declarations are in conflict and the outcome depends on cascade order.

The global `globals.scss` block sets `animation-duration: 0.01ms !important` and `animation-iteration-count: 1 !important` for all elements under `prefers-reduced-motion: reduce`. The local module override then resets these to `1.5s` and `infinite`, defeating the global rule. Result: the spinner animates continuously at 1.5s per rotation for motion-sensitive users.

**Correct fix:** Remove the `@media (prefers-reduced-motion: reduce)` block in `newTrip.module.scss` entirely. The global reset already covers it. If the intent was to keep the spinner visible (not invisible), use `opacity` or a static border color change instead.

### F-02 -- P2 -- Spinner `animation-iteration-count: infinite !important` fights global reset

**Commit:** 115fcc3 (continuation of F-01)

The local `@media (prefers-reduced-motion: reduce)` block uses `!important` on both `animation-duration` and `animation-iteration-count`, explicitly overriding the global block that does the same. Which block wins depends on which CSS module is loaded last, making this a non-deterministic race. Even in the best case the result is a slowly-rotating spinner, not a still indicator. This is a CSS-specificity trap: two `!important` declarations on the same property resolve by source order, which is not stable across builds.

### F-03 -- P2 -- Budget bar breaks visually when allocated exceeds total budget

**Commit:** 7caf8a2
**File:** `apps/client/web/src/app/(protected)/trips/[id]/page.tsx:469-496`
**User-visible symptom:** If a user selects flights + hotel + experiences that total more than their stated budget, the progress bar segments will render with combined `width` percentages exceeding 100%. Because the bar uses `overflow: hidden` on the container, the last segments are clipped rather than shown. The "Remaining" legend entry shows a negative number (e.g., "Remaining -$400") with no color change or warning -- it looks identical to a positive remaining amount except for the minus sign.

The segment widths are `(categoryTotal / budget_total) * 100` with no `Math.min(value, 100)` clamp on individual segments and no check for combined overflow. There is also no warning state (red color, "Over budget" label) when `remaining < 0`.

**Recovery path:** None surfaced in the UI. The BookingConfirmation modal does show "Over budget by $X" in `budgetNote`, which is correct, but that modal only appears after the user actively requests it. On the detail page itself there is no indication the user is over budget until they initiate the confirmation flow.

### F-05 -- P2 -- "Flexible dates" toggle is not discoverable by the user

**Commit:** d20af7f
**File:** `apps/client/web/src/components/ChatBox/TripDetailsForm.tsx:181-201` and `apps/server/src/handlers/chat/chat.helpers.ts:206-264`
**User-visible symptom:** The flexible dates toggle (Fixed / Flexible buttons) only appears in the trip details form when the backend's `buildMissingFieldsForm` includes a field with `name: 'flexible_dates'`. But `buildMissingFieldsForm` never adds that field -- it only adds destination, origin, dates, budget, and travelers. The toggle is also not rendered when the user initiates a new trip via the chat form. A user who wants flexible dates has no UI path to set them other than typing it in conversation, where the agent will infer it.

In other words: the `flexible_dates` field was wired up in `TripDetailsForm` and the server schema/repository, but the form-generation logic does not include it in the form definition it emits to the client. The feature exists in the DB and the system prompt but has no user-facing form entry point.

### F-06 -- P1 -- Instant tile selection has no undo path (US-23 acceptance criteria broken)

**Commit:** 31324d2
**File:** `apps/client/web/src/components/ChatBox/widgets/SelectableCardGroup.tsx:33-38`
**User-visible symptom:** Clicking any flight, hotel, or experience tile immediately fires `onConfirm`, persists the selection to the database via `/trips/{id}/selections`, and sends "I've selected the [X]" to the agent as a user message. The agent then advances to the next booking category. There is no intermediate "are you sure?" and no way to undo without restarting the conversation.

This violates the project CLAUDE.md rule ("Destructive / paid action that lacks a confirmation dialog is P0 regardless of how obvious it looks to the team") in spirit: while no money changes hands, the user is committing to a trip configuration item that triggers the agent to move past the current category. Changing their mind requires asking the agent in natural language, which may or may not succeed depending on agent state.

US-23 acceptance criteria explicitly state: "Clicking a card selects it (visual highlight). 'Confirm Selection' button appears. Clicking confirm sends the selection." The confirm step was deliberately removed by 31324d2. This is a documented user story that the implementation now fails.

**Mitigation in the current code:** `SelectableCardGroup` does show a green checkmark confirmation label after selection (`isConfirmed && confirmedLabel`), which communicates that the selection was recorded. But by the time the user sees it, the action has already been sent. The checkmark is post-hoc feedback, not a confirmation gate.

### F-07 -- P2 -- "Flexible" toggle default state is ambiguous (string vs. boolean)

**Commit:** d20af7f
**File:** `apps/client/web/src/components/ChatBox/TripDetailsForm.tsx:185`

The `flexible_dates` toggle reads `values.flexible_dates !== 'true'` to determine if "Fixed" is active. On initial render `values.flexible_dates` is `undefined`. So `undefined !== 'true'` is `true`, meaning "Fixed" renders as the active (pressed) state by default. This is correct behavior. However `aria-pressed={values.flexible_dates !== 'true'}` means the "Fixed" button has `aria-pressed={true}` only when `flexible_dates` is not `'true'`, which covers both the default (undefined) and the explicit 'false' states. That works.

The subtler issue: a screen reader user cycling through the form will encounter two unlabeled toggle buttons ("Fixed" and "Flexible") inside a container with no group label. The `<div className={styles.flexibleDatesToggle}>` has no `role="group"` and no `aria-labelledby` pointing to the field's `<label>`. The label element ("Flexible Dates" or whatever the agent provides) is the sibling `<label htmlFor={field.type}>` in the `.field` div, but `htmlFor="flexible_dates"` points at nothing -- there is no element with `id="flexible_dates"`. The label is visually associated by proximity but not programmatically associated. A screen reader will not announce the label when the user focuses either toggle button.

### F-09 -- P3 -- Map pin markers are not keyboard or screen reader accessible

**Commit:** d20af7f (fitBounds), e6ac1c0 (geocenter)
**File:** `apps/client/web/src/components/TripMap/TripMap.tsx:117-153`

Pin markers are created as bare `<div>` elements with inline styles, added to the Mapbox map via the Marker API. They have no `role`, no `aria-label`, and no keyboard focus. A user navigating by keyboard cannot reach or read map pins. The Mapbox popup (`setHTML("<strong>${pin.label}</strong>")`) is triggered only on click, not on keyboard focus. This is a WCAG 2.1 AA violation (1.1.1 Non-text content, 2.1.1 Keyboard).

For the current "portfolio demo" scope this is accepted risk, but it should be logged.

### F-10 -- P2 -- Budget form field enforces `min={100}` without user-visible error message

**Commit:** d20af7f
**File:** `apps/client/web/src/components/ChatBox/TripDetailsForm.tsx:223`

The budget input has `min={100}`. If a user types 50 and submits, the native browser `type="number"` validation will block submission with a browser-default tooltip ("Value must be greater than or equal to 100"). That tooltip is styled by the browser, not the app, and does not match the app's design. On some mobile browsers the native number validation tooltip is not displayed at all, silently blocking the submit. There is no app-level inline error message explaining the minimum.

### F-12 -- P2 -- Mobile tab bar lacks ARIA tab pattern

**File:** `apps/client/web/src/app/(protected)/trips/[id]/page.tsx:348-361`
**User-visible symptom:** The Chat / Itinerary switcher on mobile is implemented as two plain buttons toggling display state. A screen reader user will hear "Chat, button" and "Itinerary, button" with no indication that these are tabs controlling visible panel content. The panes themselves have no `role="tabpanel"` and no `aria-labelledby` linking them back to their controlling tab. A blind user cannot tell which pane is currently active without reading its contents.

The correct pattern: `role="tablist"` on the container, `role="tab"` with `aria-selected` on each button, `role="tabpanel"` with `aria-labelledby` on each pane.

### F-16 -- P3 -- "Thinking..." indicator and pending dots animation are both shown simultaneously during streaming

**File:** `apps/client/web/src/components/ChatBox/VirtualizedChat.tsx:173-193`

During an active agent turn, both the `pendingIndicator` (ChatProgressBar in indeterminate mode) and the `thinkingIndicator` (three bouncing dots + "Thinking..." text) can render at the same time. The pendingIndicator renders when `isSending && streamingNodes.length === 0 && toolProgress.length === 0 && streamingText === ''`. The thinkingIndicator renders when `isStreaming` (passed as the same value as `isSending`). For the brief window at the start of a turn before the first tool progress event arrives, both indicators are visible simultaneously, creating visual noise.

### F-17 -- P2 -- No loading state or timeout feedback for the 15s SerpApi timeout

**Commit:** d20af7f (`serpApiGet` now has `AbortSignal.timeout(15_000)`)
**File:** `apps/server/src/services/external/serpapi.service.ts:69`

The timeout is a real fix: the 15s `AbortSignal.timeout` prevents the fetch from hanging indefinitely. When it fires, the error is caught by `flights.tool.ts` and an empty array is returned -- the agent then narrates "no flights found." This is a valid fail-soft. However, from the user's perspective, they watch a spinning ChatProgressBar ("Searching flights") for up to 15 seconds and then get a natural-language "I couldn't find flights" -- with no indication of whether this is because flights don't exist, the API timed out, or the quota was hit. The three failure modes are indistinguishable to the user.

This is not a blocker -- the P0 freeze is fixed -- but the user experience of a 15-second wait followed by an opaque "no flights" message is poor.

### F-18 -- P3 -- BookingConfirmation has two consecutive `h2` elements with different content

**File:** `apps/client/web/src/components/BookingConfirmation/BookingConfirmation.tsx:98-106`

The modal renders:
- `<h2>Save your itinerary for</h2>` (inside the image overlay)
- `<h2>Save Your Itinerary for {destination}</h2>` (visible below the image)

Both are visible to screen readers (the image overlay h2 is not `aria-hidden`). A screen reader user hears the same headline twice consecutively, which is confusing. The first h2 (inside the image) should be `aria-hidden="true"` since it is decorative reinforcement of the second.
