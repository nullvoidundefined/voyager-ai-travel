# Voyager UX Audit

**Date:** 2026-04-06
**Auditor:** Chief Experience Officer (canonical role: `~/.claude/audits/ux.md`)
**Scope:** Voyager web client, user stories US-1 through US-35, chat agent UX, booking flow, preferences wizard, a11y, mobile, destructive action guardrails.

---

## Executive Summary

**Can a new user complete the primary happy path (register → preferences → create trip → chat with agent → confirm booking) without help?**

**No.** A determined technical user can get through it, but a non-technical first-time user will stall in at least two places. Specifically: (1) `/trips/new` spins up an empty trip with `destination: "Planning..."` and dumps the user into a chat with no visible "here is how to talk to me" coaching beyond the form, and (2) the booking confirmation modal's `Confirm Booking` button starts a hard-coded 2.2s "booking" animation that cannot be cancelled and which is the only guardrail on a nominally paid action. A real user who mis-clicks has no way out.

**Top 3 priorities (P0 / P1):**

1. **P0. Destructive action guardrails missing.** Two user-visible actions take irreversible effect with no confirmation dialog: trip deletion (`web-client/src/app/(protected)/trips/page.tsx:112-138`: click the `×` on a trip card and the mutation fires with zero prompt, via an optimistic cache wipe) and trip booking (`web-client/src/components/BookingConfirmation/BookingConfirmation.tsx:75-87`: clicking `Confirm Booking` enters a `booking` state for 2.2s, then auto-advances to `confirmed` and calls `onConfirm` after another 1.5s, with no `Cancel` affordance during that window). The canonical role explicitly flags this as a P0 regardless of how "obvious" it looks to the team. US-16 and US-27 both need proper confirmation.
2. **P1. User story coverage gap: 33 of 35 stories have zero E2E coverage.** Only `e2e/auth.spec.ts` and `e2e/navigation.spec.ts` exist. Every story in Trip Management, Chat & Booking, Checkout, Preferences, and Account is untested end-to-end. `docs/USER_STORIES.md:520-531` even names the missing files (`trips.spec.ts`, `chat.spec.ts`, `checkout.spec.ts`, `preferences.spec.ts`, `account.spec.ts`, `public.spec.ts`, `explore.spec.ts`). Per the canonical role, every documented user story must have an E2E test.
3. **P1. Chat error recovery is unactionable and silent.** When the SSE stream fails mid-agent-loop, `web-client/src/components/ChatBox/useSSEChat.ts:135-139` sets `streamingText` to the literal string `"Something went wrong. Please try again."` and drops the message. The canonical role names this exact failure mode. There is no retry button, no indication which tool call failed, no way to recover the partial itinerary the user was building. The same file at line 73-75 handles `error` SSE events by overwriting `streamingText` with the raw error, which is a different but related anti-pattern (raw error leaked into UI, contradicting the CLAUDE.md Toast convention).

## Critical Path Analysis: Signup → First Value → Paid Action

**Step 1: Signup.** `(auth)/register`. Unverified, but US-8 says a successful register opens the Preferences Wizard. **Friction risk:** no password strength indicator visible in the component tree. **Dead end risk:** if the wizard fails to open (e.g., preferences endpoint 500s post-register), the user lands on an empty state with no clear next step.

**Step 2: Preferences wizard.** 6 steps. Time-to-value is acceptable (skip button at `PreferencesWizard.tsx:297-301` lets users bypass everything). However: the wizard does not explain **what the product does** before asking personal questions (accommodation style, dining preferences, LGBTQ+ safety, gender). A first-time user has not yet seen Voyager plan a trip and is being asked deeply personal travel-pattern questions. **Recommendation: add a one-sentence "why we ask" on the wizard header**, and consider making the wizard skippable in one click ("Skip for now, I'll set these later") rather than requiring per-step skip.

**Step 3: Create trip.** `trips/new/page.tsx` creates a trip with `destination: "Planning..."` and redirects. The literal string `"Planning..."` shows up on the trips list card (`trips/page.tsx:142`: `<h2>{trip.destination}</h2>`) and the detail page hero. **This is a bug-class UX issue**: the user sees a literal status word where they expect a city name.

**Step 4: First value (chat with agent).** User arrives at `trips/[id]` with an empty chat. `ChatBox.tsx` does not render a client-side welcome message. It depends on the server having seeded one, or on `TripDetailsForm` rendering. **Risk:** if the server-side welcome is missing or delayed, the user sees an empty chat with a placeholder "Ask the agent to plan your trip..." and no coaching. **Recommendation:** render a static welcome message client-side if `serverMessages` is empty and no form is visible.

**Step 5: Paid action (confirm booking).** Covered in Destructive Actions below. **This is the single biggest trust-breaking moment in the app and it is not safeguarded.**

## Error Recovery

Every failure mode a user could hit:

- **SSE stream aborts mid-response.** `useSSEChat.ts:135-139`: sets `streamingText` to `"Something went wrong. Please try again."`. **P1. Unactionable.** No retry button, no indication of what failed (which tool? which step?), no way to resume the agent loop. The partial streaming nodes that were rendered get wiped in the `finally` block (line 158-161).
- **Server returns an `error` SSE event.** `useSSEChat.ts:73-75`: overwrites `streamingText` with the raw `event.error` string. **P1. Leaks raw error to user.** Contradicts the project's own CLAUDE.md convention: "Use Toast component for API/server errors. Never show raw error messages or stack traces inline."
- **Tool call fails (e.g., SerpApi rate limit, Google Places empty).** No evidence the tool_progress indicator surfaces a "this tool failed" terminal state to the user. `ToolProgressIndicator` is in the node list but not inspected; verify it has an `error` status variant and that the agent's follow-up reasoning is surfaced ("I couldn't find flights, trying a different date range").
- **User changes their mind mid-plan.** No undo. `ChatBox.tsx:198-209` offers a `Try Again` button after flights land, but this only appends a new message asking the agent to reconsider. There is no way to **remove** an already-selected tile card from the trip once it is confirmed, short of asking the agent conversationally. `SelectableCardGroup` is the only selection primitive, and once "Confirm Selection" fires, the selection is persisted server-side with no client-side undo.
- **User deletes a trip by accident.** No confirmation, no undo. P0. See Destructive Actions.
- **User clicks `Confirm Booking` by accident.** 2.2s animation, no cancel, auto-closes. P0. See Destructive Actions.
- **Network failure creating a new trip.** `trips/new/page.tsx:30-33` sets `error` state and shows a "Failed to start a new trip. Please try again." message with no retry button. User must navigate back to `/trips` and click `+ New Trip` again.
- **Network failure loading trip detail.** `trips/[id]/page.tsx:115-124` shows "Trip not found." regardless of whether it is a 404 or a 500. **P2. Misleading error.** A 500 is not a "not found".

## Destructive Action Guardrails

**This is the most critical section of the audit.** Per the canonical role, every destructive or paid action must have a confirmation dialog. Two do not.

### P0-1: Trip deletion has no confirmation

`web-client/src/app/(protected)/trips/page.tsx:112-138`

```tsx
<button
  type='button'
  className={styles.deleteBtn}
  aria-label='Delete trip'
  onClick={(e) => {
    e.preventDefault();
    deleteMutation.mutate(trip.id);
  }}
>
  &times;
</button>
```

One click on the `×`, the trip is gone. Optimistic cache update (`onMutate` at line 60-66) removes it from the UI immediately, and there is no undo snackbar. A user reaching toward an adjacent card on a phone can wipe a trip they spent 20 minutes planning. **Required fix:** wrap in a confirmation dialog (`<AlertDialog>` or similar), OR implement an "undo" toast with 5-second window that re-inserts from the rolled-back cache. Either is acceptable; neither is present.

### P0-2: Trip booking has no real confirmation

`web-client/src/components/BookingConfirmation/BookingConfirmation.tsx:75-87`

```tsx
useEffect(() => {
  if (stage === 'booking') {
    const timer = setTimeout(() => setStage('confirmed'), 2200);
    return () => clearTimeout(timer);
  }
}, [stage]);

useEffect(() => {
  if (stage === 'confirmed') {
    const timer = setTimeout(onConfirm, 1500);
    return () => clearTimeout(timer);
  }
}, [stage, onConfirm]);
```

The flow is: user clicks `Confirm Booking` → modal enters `booking` state → 2200ms spinner → `confirmed` state → 1500ms "You're going to X" → `onConfirm` fires, which calls `handleConfirmBooking` in `trips/[id]/page.tsx:90-101` which PUTs `status: 'saved'`. **There is no way to cancel this sequence once it starts.** The `Make Changes` button only exists in the `review` stage. The design is "press one button, trip is booked in 3.7 seconds". This is _technically_ a confirmation modal because the user had to open it via `Confirm booking`, but the canonical role's requirement is stricter: **every destructive or paid action must have a confirmation**. In this design, the modal itself is the confirmation. And the button inside it says `Confirm Booking`. But there is no "are you sure" second step for the irreversible transition. Combined with the fact that the trip enters a locked `saved` state that disables the chat (`ChatBox.tsx:229`), this is a one-way door.

**Required fix:** The `review` stage already shows an itemized breakdown (good, US-26). But the `Confirm Booking` button needs to either (a) be a two-step press ("Confirm → Really confirm"), or (b) keep a visible "Cancel" option during the `booking` spinner stage, or (c) offer an "unbook / edit" path from the booked trip detail page for at least a short grace period.

Also note this is a **mock booking**: nothing actually gets charged. But the UX treatment should still meet the guardrail because real money will flow when this is wired to a payment processor and the UX patterns will carry over.

### Other destructive actions

- **Logout** (`Header.tsx:126-128`): no confirmation. Acceptable. Logout is reversible (log back in).
- **Wizard skip** (`PreferencesWizard.tsx:297-301`): no confirmation. Acceptable. User can return via account page.
- **Overlay click on PreferencesWizard** (`PreferencesWizard.tsx:181-185`): closes without confirming unsaved state. **P3**: minor; user can re-open.
- **Tile card selection confirm** (`SelectableCardGroup`): confirms and persists to trip. Reversible only via asking the agent. **P2**: needs an "unselect" affordance.

## Cognitive Load

- **Chat + tile cards + quick replies + form + booking actions all live in one ChatBox.** `ChatBox.tsx` renders up to five distinct interaction surfaces simultaneously (streaming text, tool progress, tile cards, quick replies, booking action buttons, and the input). At agent-reply time the user may see: optimistic user bubble + tool progress nodes + streaming text + hotel tile cards + quick reply chips + booking action buttons. **P2. Decision fatigue risk.** Consider collapsing tool progress into a dismissable drawer once the agent finishes.
- **Terminology:** "trip" vs "itinerary" vs "booking" are used interchangeably. `BookingConfirmation.tsx` says "Booking" and "Confirm your trip"; `trips/[id]/page.tsx` has `bookedBadge` with text "Booked". Consistent-ish.
- **"Planning..." as a destination name** (see US-14): this is a literal placeholder that looks like a status word. Extreme confusion risk on the trips list. **P1.**
- **Preferences wizard jargon:** "Budget Comfort", "Travel Pace", "LGBTQ+ safety". Mostly clear but `budget_comfort` has no in-wizard explanation of what the values mean without opening the step component.

## Consistency & Patterns

- **Loading treatments inconsistent** across routes (plain text vs spinner).
- **Error treatments inconsistent**: toast convention is documented in project CLAUDE.md but `useSSEChat.ts:74` writes the raw error into `streamingText`, and `trips/page.tsx:89-91` renders the error inline as text.
- **Trip status terminology**: DB status is `planning | saved | archived` (per `trips/page.tsx:19`), but the user-facing word for `saved` is "Booked" (`bookedBadge`). Inconsistent terminology between data layer and UI. Low-impact but worth noting.
- **Radix UI:** per CLAUDE-FRONTEND conventions, all modals/dialogs should use Radix primitives. `BookingConfirmation.tsx` and `PreferencesWizard.tsx` both hand-roll the overlay/modal pattern. **P1. Both should be migrated to Radix `<Dialog>`/`<AlertDialog>`** to inherit focus trap, Escape handling, aria attributes, and focus return for free.
